## [슬라이드 05] 왜 설정값을 코드에 박아두면 안 될까

"만약 이미지 안에 설정값을 하드코딩해버리면 어떨까요? 값을 하나 바꿀 때마다 이미지를 다시 빌드해야
합니다. 3주차에 배운 대로 이미지 빌드는 시간이 걸리는 작업이죠.

게다가 같은 이미지를 개발 환경과 운영 환경에서 서로 다른 설정으로 쓰고 싶을 때도 곤란해집니다.

**(해결책 제시)** 그래서 K8s는 **ConfigMap**이라는, 설정값을 이미지 밖에 따로 저장해두는 오브젝트를
제공합니다. 핵심 아이디어는 — 이미지(코드)는 그대로 두고, ConfigMap만 바꿔서 다른 환경에 재사용하는
겁니다."

---

## [슬라이드 06] ConfigMap 사용법

"`kubectl create configmap {이름} --from-literal=KEY=VALUE`처럼 만들고, Pod의 환경변수로 이 값을
불러와서 씁니다.

로그 레벨, API 서버 주소, 최대 연결 수처럼 '민감하지 않은' 설정값에 주로 사용합니다."

## [슬라이드 08] Secret = ConfigMap의 '보안 버전'

"사용법은 ConfigMap과 거의 똑같습니다. 다만 Secret은 값이 Base64로 인코딩되어 저장되고, 접근 권한을
더 엄격하게 통제할 수 있습니다. DB 비밀번호, API 키, 인증서 같은 민감한 정보에 씁니다.

**(중요한 오해 방지)** 여기서 꼭 짚어야 할 게 있는데, Base64는 **암호화가 아닙니다.** 그냥 인코딩
방식이라서 누구나 디코딩할 수 있어요. 진짜 보안은 '누가 이 Secret에 접근할 수 있는가'를 관리하는
권한 설정으로 확보하는 겁니다. 오늘은 Secret의 '사용법'까지만 다루고, 접근 권한 관리는 이 과목의
범위를 넘어서니 참고로만 알아두세요."

## [슬라이드 09] Killercoda — Secret 환경변수 실습

"자, 직접 해보겠습니다. 교수자가 미리 만들어 둔 Secret이 클러스터에 있습니다. `kubectl get secret`
으로 확인해보세요.

그다음 그 Secret 값을 환경변수로 불러오는 Pod YAML을 작성해서 `kubectl apply`하고, `kubectl exec`로
Pod 안에 들어가 `env` 명령으로 그 값이 제대로 들어왔는지 확인하세요.

**(화면 공유해 실제로 시연)** 제가 먼저 보여드릴게요."

## [슬라이드 10] VM 안내

"개인 VM에서 `kubectl create configmap`으로 간단한 설정값을 하나 만들고, `kubectl create secret
generic`으로 비밀값도 하나 만들어보세요.

`kubectl get configmap`, `kubectl get secret`으로 확인한 결과를 캡처해서 제출하시면 됩니다."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "Secret이 안전하지 않다면 왜 쓰나요?" | Base64 인코딩 자체는 암호화가 아니지만, K8s의 접근 권한 관리(RBAC)와 결합하면 "이 Secret은 특정 Pod·사용자만 읽을 수 있다"처럼 통제할 수 있습니다. 평문으로 환경변수에 직접 쓰는 것보다는 훨씬 안전한 관리 방법입니다. |
| "ConfigMap 값을 파일로도 넣을 수 있나요?" | 네, --from-file 옵션으로 설정 파일 전체를 ConfigMap으로 만들 수도 있습니다. 오늘은 --from-literal(간단한 key=value)만 다룹니다. |
| "값을 바꾸면 실행 중인 Pod에 바로 반영되나요?" | 기본적으로는 아닙니다. Pod를 재시작해야 새 값이 반영되는 경우가 많습니다. 이 과목에서는 다루지 않는 세부 동작입니다. |
