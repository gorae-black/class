## [슬라이드 05] Pod YAML에 PVC 붙이기

"방법은 간단합니다. Pod YAML의 `volumes`에 지난 시간에 만든 PVC 이름을 지정하고, 컨테이너의
`volumeMounts`로 그 볼륨을 컨테이너 안 특정 경로에 연결하면 됩니다.

오늘은 MySQL이 실제로 데이터를 저장하는 경로인 `/var/lib/mysql`에 연결할 겁니다. 이 순간부터, MySQL
컨테이너 안에서는 그냥 평범한 로컬 디렉토리에 저장하는 것처럼 보이지만, 실제로는 우리가 지정한
hostPath 디렉토리에 데이터가 저장되는 거예요."

## [슬라이드 07] MySQL Pod 배포하기

"공식 mysql 이미지를 사용합니다. 환경변수로 `MYSQL_ROOT_PASSWORD`를 지정해서 루트 비밀번호를
설정하고요.

**(1차시 연결)** 여기서 1차시에 배운 Secret을 활용하면 더 안전하게 비밀번호를 관리할 수 있습니다.
비밀번호를 YAML에 평문으로 쓰는 대신 Secret에서 불러오는 방식이죠. 오늘 실습에서는 간단하게 환경변수로
직접 지정하지만, 실무에서는 Secret을 쓰는 게 표준입니다.

그리고 앞서 배운 대로 PVC를 `/var/lib/mysql`에 마운트합니다.

**(정리 포인트)** 자, 이 시점에서 보면 6주차에 배운 내용이 전부 하나로 연결되고 있죠? 1차시의
Secret(비밀번호 관리)과 2차시의 PV/PVC(저장공간)가 오늘 3차시의 MySQL 실전 배포에서 함께 쓰이고
있습니다."

---

## [슬라이드 08] 영속성 검증 절차

"검증 절차는 세 단계입니다.

첫째, MySQL Pod에 접속해서 테이블을 하나 만들고 데이터를 한 줄 넣습니다.
둘째, `kubectl delete pod`로 그 Pod를 강제로 삭제합니다. Deployment가 있다면 자동으로 새 Pod를
만들어줄 거예요.
셋째, 새로 생성된 Pod에 접속해서 아까 넣은 데이터가 SELECT로 그대로 조회되는지 확인합니다.

**(핵심 포인트)** 만약 PVC 없이 그냥 MySQL을 배포했다면, Pod가 삭제되고 새로 생기는 순간 데이터가
전부 사라졌을 거예요. 오늘 우리는 PVC 덕분에 데이터가 살아남는 걸 직접 확인하는 겁니다."

---

## [슬라이드 09] Killercoda — MySQL 배포+영속성 검증

"자, 직접 해보겠습니다.

mysql.yaml을 작성하세요. 지난 2차시에 만든 PVC를 `/var/lib/mysql`에 마운트하는 걸 잊지 마세요.
`kubectl apply`로 배포한 다음, `kubectl exec`로 Pod에 접속해서 mysql 클라이언트로 테이블 생성과 데이터
삽입을 해보세요.

그다음 `kubectl delete pod {mysql파드이름}`을 실행하세요.

**(주의사항 강조)** 여기서 중요한 안내입니다 — 새 Pod가 완전히 Running 상태가 되기까지 **10~20초
정도** 걸립니다. `kubectl get pods`로 상태를 확인하면서 기다려주세요. 너무 빨리 접속을 시도하면 아직
컨테이너가 준비되지 않아서 오류가 날 수 있습니다.

새 Pod가 준비되면 다시 접속해서 SELECT로 데이터가 그대로인지 확인하세요.

**(화면 공유해 실제로 시연)** 제가 먼저 보여드릴게요."

## [슬라이드 10] VM 안내 및 6주차 종합 제출

Pod 삭제 **전**의 SELECT 결과와, 삭제 **후** 새 Pod에서의 SELECT 결과를 모두 캡처해서 데이터가

**(주차 마무리 안내)** 오늘이 6주차 마지막 시간이니, 1차시(ConfigMap/Secret)·2차시(PV/PVC)·3차시(오늘,
MySQL 영속성 검증) 실습 결과를 한 번에 정리해서 LMS 과제 게시판에 제출해주세요."

## [슬라이드 11] 오늘의 정리

"오늘 배운 내용, 두 가지로 정리하겠습니다.

첫째, **PVC를 Pod의 volumeMounts에 연결하면 컨테이너 데이터가 hostPath에 실제로 저장된다.**
둘째, **Pod가 재생성되어도 PV에 저장된 데이터는 그대로 유지된다.**"

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "Pod를 삭제했는데 새 Pod가 안 생겨요" | Deployment로 배포했는지 확인이 필요합니다. 단순 Pod로만 배포했다면 삭제 시 자동 재생성이 안 됩니다. 4주차에서 배운 대로 Deployment는 원하는 개수를 항상 유지하려 하기 때문에 자동으로 다시 만들어줍니다. |
| "10~20초씩 왜 걸리나요?" | 새 컨테이너를 만들고 MySQL 프로세스가 초기화되는 시간입니다. 특히 MySQL은 시작 시 내부적으로 여러 초기화 작업을 하기 때문에 nginx 같은 가벼운 이미지보다 시간이 더 걸립니다. |
| "실무에서도 hostPath를 그대로 쓰나요?" | 아니요. hostPath는 노드 하나에 데이터가 묶여있어서, 그 노드가 아예 사라지면 데이터도 함께 사라집니다. 실무에서는 클라우드 스토리지(AWS EBS 등)나 네트워크 스토리지를 PV로 사용해 노드와 무관하게 데이터를 보존합니다. 오늘은 로컬 실습 환경이라 hostPath로 원리만 익힌 것입니다. |
