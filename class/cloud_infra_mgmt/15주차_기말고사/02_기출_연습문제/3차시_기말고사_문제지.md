## Part 1. 아키텍처 분석·설계 서술형 (50점)

**1. (10점)** 9~10주차에서 배운 로컬 State와 원격 State(Terraform Cloud)의 차이를 설명하고, 왜
협업 환경에서 원격 State가 필요한지 서술하시오.

**2. (10점)** 12주차에서 GitHub의 Webhook 방식 대신 Poll SCM 방식을 사용한 이유를 설명하시오.

**3. (15점)** 13주차 파이프라인에서, Jenkins 컨테이너가 minikube에 직접 kubectl로 접속하지 못하는
이유를 설명하고, 이를 우회하기 위해 실제로 사용한 방법을 서술하시오.

**4. (15점)** 14주차에서 배운 HPA(Scale-out)와 자가치유(Self-Healing)가 함께 동작했을 때 얻을 수
있는 이점을, "고가용성"이라는 개념과 연결해 서술하시오.

## Part 2. 코드 트러블슈팅 (50점)

**5. (25점)** 아래는 학생이 작성한 Jenkinsfile의 Push 단계입니다. 이 코드에는 문제가 있습니다.
무엇이 문제이고, 왜 문제인지, 그리고 어떻게 고쳐야 하는지 서술하시오.

```groovy
stage("Push") {
  steps {
    sh "docker login -u myuser -p SuperSecret123!"
    sh "docker push myuser/chatbot:${BUILD_NUMBER}"
  }
}
```

**6. (25점)** 학생이 아래와 같이 HPA를 설정하고 `ab`로 부하 테스트를 진행했지만, 부하를 아무리
줘도 Pod 개수가 전혀 늘어나지 않았습니다. 아래 설정에서 원인이 되는 부분을 찾아 설명하고, 어떻게
수정해야 하는지 서술하시오. (참고: 실습에 사용한 VM은 2 vCPU입니다.)

```
kubectl set resources deployment chatbot -c=web --requests=cpu=2000m,memory=512Mi
kubectl autoscale deployment chatbot --cpu-percent=50 --min=1 --max=5
```
