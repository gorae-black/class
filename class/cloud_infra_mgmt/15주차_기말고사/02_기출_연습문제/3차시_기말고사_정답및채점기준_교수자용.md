문제지: [`3차시_기말고사_문제지.md`](3차시_기말고사_문제지.md) (총 100점)

## Part 1. 아키텍처 분석·설계 서술형 (50점)

### 1번 (10점)
- **모범 답안**: 로컬 State는 Terraform을 실행한 개인의 VM 디스크에만 저장되어, 여러 사람이 같은
  인프라를 다룰 때 서로 다른 State를 갖게 되어 충돌이 발생할 수 있다. 원격 State(Terraform Cloud)는
  하나의 공유된 저장소에 State를 두어, 누가 작업하든 항상 동일한 최신 State를 참조하게 함으로써
  충돌을 방지한다.
- **채점**: 로컬/원격 State의 차이 설명 5점 + 협업 시 원격 State가 필요한 이유(충돌 방지, 일관성
  유지) 5점

### 2번 (10점)
- **모범 답안**: 학생 개인 VM은 가정용 공유기의 NAT 뒤에 있어 외부(GitHub)에서 직접 접속할 수 있는
  공인 주소가 없다. Webhook은 GitHub이 Jenkins 서버로 직접 접속해서 알려주는 방식이라 이 환경에서
  동작하지 않는다. 반면 Poll SCM은 Jenkins가 스스로 주기적으로 GitHub에 접속해 변경 여부를
  확인하는 방식이라, Jenkins 쪽에서 외부로 나가는 연결만 있으면 되므로 NAT 환경에서도 동작한다.
- **채점**: NAT/공인 주소 부재 문제 설명 5점 + Poll SCM이 이 문제를 해결하는 원리 설명 5점

### 3번 (15점)
- **모범 답안**: Jenkins와 minikube는 각각 별도의 Docker 컨테이너로 떠 있으며, 서로 다른 Docker
  네트워크에 속해 있거나 IP 충돌 등의 문제로 Jenkins 컨테이너에서 minikube 클러스터로 직접 네트워크
  연결이 되지 않는다. 이를 우회하기 위해, Jenkins 컨테이너에 마운트된 `/var/run/docker.sock`을
  이용해 `docker exec`로 minikube 컨테이너 **내부에** 이미 설치되어 있는 kubectl을
  `--kubeconfig=/etc/kubernetes/admin.conf` 옵션과 함께 실행하는 방식을 사용했다.
- **채점**: 직접 접속이 안 되는 이유 설명 7점 + docker exec 우회 방법 설명 8점

### 4번 (15점)
- **모범 답안**: HPA(Scale-out)는 트래픽이 몰릴 때 Pod 개수를 자동으로 늘려 부하를 분산시켜
  서비스가 느려지거나 다운되는 것을 막는다. 자가치유(Self-Healing)는 Pod가 장애로 죽더라도 즉시
  새 Pod를 만들어 개수를 유지한다. 이 둘이 함께 동작하면, '트래픽 폭주'라는 위기와 '장애 발생'이라는
  위기 모두에서 사람의 개입 없이 서비스가 스스로 대응하고 살아남을 수 있다 — 이것이 고가용성(High
  Availability)이다.
- **채점**: HPA의 역할 설명 5점 + 자가치유의 역할 설명 5점 + 둘이 결합했을 때의 고가용성 의미 설명
  5점

## Part 2. 코드 트러블슈팅 (50점)

### 5번 (25점)
- **문제점**: `docker login` 명령에 Docker Hub 비밀번호(`SuperSecret123!`)가 **평문으로 그대로
  노출**되어 있다. 이 Jenkinsfile이 GitHub 저장소에 올라가면, 저장소에 접근할 수 있는 누구나 이
  비밀번호를 볼 수 있게 되어 심각한 보안 문제가 된다.
- **왜 문제인가**: 12주차에서 배운 대로, Jenkinsfile은 코드로 관리되며 보통 GitHub에 함께 저장된다.
  비밀번호를 코드에 그대로 적으면 버전 관리 이력에도 영구히 남아, 나중에 지워도 과거 커밋에서 계속
  노출된다.
- **수정 방법**: `withCredentials([usernamePassword(credentialsId: "dockerhub-cred",
  usernameVariable: "USER", passwordVariable: "PASS")]) { sh "echo $PASS | docker login -u $USER
  --password-stdin" }`처럼 Jenkins Credentials에 미리 등록해둔 값을 안전하게 불러와 사용해야 한다.
- **채점**: 문제점(평문 노출) 지목 10점 + 왜 문제인지(보안 위험, Git 이력에 영구 노출) 설명 8점 +
  올바른 수정 방법(withCredentials) 7점

### 6번 (25점)
- **문제점**: `--requests=cpu=2000m`으로 설정된 부분이 원인이다. `2000m`은 2 코어(2000 밀리코어)를
  의미하는데, 실습에 사용하는 VM 자체가 2 vCPU다. 즉 Pod 하나가 요청하는 자원만으로도 VM 전체 CPU와
  맞먹어서, 실제 사용률이 request 대비 50%(HPA의 목표치)에 도달하는 것이 사실상 불가능하거나 매우
  어렵다.
- **왜 문제인가**: HPA는 "실제 CPU 사용량"을 "resource request 대비 비율(%)"로 계산한다. request가
  너무 높게 설정되면 그 값 자체가 임계값을 사실상 도달 불가능하게 만들어, 아무리 부하를 줘도
  스케일업이 관찰되지 않는다.
- **수정 방법**: `--requests=cpu=50m`처럼 request를 낮게 설정해야, 적은 부하로도 request 대비 사용률
  비율이 쉽게 목표치(50%)를 넘어서 HPA가 정상적으로 Pod를 늘릴 수 있다.
- **채점**: 원인이 되는 부분(cpu=2000m) 지목 10점 + HPA의 계산 방식(request 대비 비율)과 연결한
  원인 설명 10점 + 올바른 수정 방향(낮은 request 값) 5점
