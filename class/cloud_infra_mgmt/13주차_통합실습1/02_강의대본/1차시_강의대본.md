## [슬라이드 05] 왜 오늘부터 Killercoda가 없을까

"Killercoda는 60분 시간 제한이 있었죠. 그런데 오늘부터 하게 될 실습은 여러 도구가 복잡하게
연결되는 파이프라인이라서, 문제가 생기면 디버깅에 시간이 꽤 걸릴 수 있습니다. 60분 안에 끝내야
한다는 압박이 있으면 오히려 학습에 방해가 되겠죠.

**(안내)** 그래서 13~14주차는 100% 학생 개인 VM 안에서만, 시간 제약 없이 진행합니다.

**(미션 제시)** 오늘부터 우리의 미션은 이겁니다 — AI 미니 챗봇 웹 앱을 Git → Jenkins → Docker Hub
→ K8s로 완전 자동 배포하고, 다음 주(14주차)에는 그 서비스가 부하를 견디게 만드는 것까지 해볼
겁니다."

## [슬라이드 07] 13주차에 완성할 파이프라인

"화면의 그림을 보시면 이렇습니다.

학생 PC 에디터에서 소스를 수정하고 `git push origin main`을 실행합니다. 그러면 로컬 VM의
Jenkins가 12주차에 배운 Poll SCM(매 1분)으로 이 변경을 감지합니다. Jenkins는 Docker 이미지를
빌드하고 Docker Hub에 Push합니다. 마지막으로 Jenkins가 minikube의 Deployment 이미지를 새 버전으로
교체합니다 — 이걸 Rolling Update라고 부릅니다.

**(전체 그림 강조)** 이 네 단계, '학생 PC → GitHub → Jenkins → Minikube'가 오늘부터 우리가 만들
전체 파이프라인입니다."

---

## [슬라이드 08] 각 도구가 맡는 역할 한 줄 정리

"각 도구가 이 파이프라인에서 어떤 역할을 맡는지, 지금까지 배운 것과 연결해서 정리해보겠습니다.

**Git/GitHub**(9~10주차에서 배웠죠) — 소스 코드의 버전을 기록하고 변경을 공유합니다.
**Docker**(1~3주차) — 챗봇 앱을 이미지로 포장해서 어디서든 동일하게 실행되게 합니다.
**Jenkins**(11~12주차) — 코드 변경을 감지해서 빌드·배포를 자동으로 실행합니다.
**Kubernetes**(4~7주차) — 실제로 챗봇 컨테이너를 운영하고, 새 버전으로 무중단 교체합니다.

**(핵심 메시지)** 지금까지 따로따로 배웠던 이 네 가지가, 오늘부터 하나의 파이프라인 안에서 서로
맞물려 돌아가는 걸 직접 경험하게 될 거예요."

## [슬라이드 09] 미니 챗봇이란?

"오늘 배포할 대상, '미니 챗봇'을 소개해드릴게요.

Flask로 만든 간단한 웹 앱인데, 사용자가 메시지를 보내면 규칙 기반으로 응답하는 미니 챗봇입니다.
복잡한 AI 모델이 아니라, '이 파이프라인이 무엇을 배포하는지' 보여주기 위한 실습용 앱이에요.

**(초점 명확화)** 오늘 실습의 핵심은 챗봇 자체의 성능이 아니라, 이 앱이 **자동으로 빌드·배포·업데이트되는 전체 과정**입니다. `04_템플릿코드`에 챗봇 뼈대 코드(`app.py`, `Dockerfile`,
`deployment.yaml`)를 미리 준비해두었습니다."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "챗봇에 진짜 AI(예: GPT)를 연결해도 되나요?" | 이 과목의 필수 실습 범위는 아니지만, 관심 있는 학생은 선택 심화로 시도해볼 수 있습니다. 다만 외부 API 키 관리, 비용 문제가 따르므로 필수 제출물에는 포함하지 않는다고 명확히 안내하세요. |
| "Rolling Update가 정확히 뭔가요?" | K8s Deployment가 새 이미지로 Pod를 교체할 때, 기존 Pod를 한꺼번에 지우지 않고 하나씩 순차적으로 교체하는 방식입니다. 그래서 교체 도중에도 서비스가 완전히 끊기지 않습니다. 2차시에서 직접 명령어로 확인합니다. |
| "오늘 실습을 못 끝내면 어떻게 하나요?" | 오늘은 흐름도를 그려보는 예습 성격의 시간이라 부담을 갖지 않아도 됩니다. 다만 다음 시간부터는 실제 구축이 시작되니, 오늘 자료(챗봇 코드 구조)는 꼭 한 번 훑어보고 오도록 안내하세요. |
