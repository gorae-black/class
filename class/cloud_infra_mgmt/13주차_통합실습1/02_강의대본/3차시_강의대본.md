## [슬라이드 05] 지금까지는 '조립'했다면, 오늘은 '작동 확인'

"2차시까지는 Build Now 버튼을 눌러서 각 단계가 개별적으로 성공하는지 확인했습니다. 말하자면
자동차의 각 부품이 따로따로 잘 작동하는지 본 거예요.

**(오늘의 의미)** 오늘은 사람이 버튼을 누르지 않고, **코드 수정 → git push**만으로 전체가 저절로
돌아가는지 확인합니다. 이 흐름이 실제로 성공하면, 자동차 전체가 시동을 걸고 굴러가는 걸 확인하는
셈이죠 — 진짜 자동 배포 파이프라인이 완성되는 순간입니다.

**(오늘의 최종 목표)** 오늘의 목표는 명확합니다 — `app.py`의 챗봇 응답 문구 하나를 바꾸고
push했을 때, 실제 챗봇의 응답이 바뀌는 것까지 직접 확인하는 겁니다."

## [슬라이드 07] Stage별 자주 발생하는 문제

"화면의 표를 보시면, Stage별로 자주 생기는 문제를 정리했습니다.

**Checkout**에서 막히면 대부분 저장소 URL 오타나, 저장소가 Private으로 설정된 경우입니다.
**Build**에서 막히면 Dockerfile 문법 오류나 이미지 이름 오타가 흔합니다.
**Push**에서 막히면 `dockerhub-cred`를 등록하지 않았거나 Docker Hub 아이디를 잘못 입력한 경우입니다.
**Deploy**에서 막히면 Deployment나 컨테이너 이름이 실제 YAML과 다르거나, minikube가 꺼져있는
경우가 많습니다."

## [슬라이드 08] 진단 순서 — Console Output부터

"막혔을 때 진단하는 순서를 알려드릴게요.

첫째, Jenkins Job의 **Console Output**에서 빨간 글씨(오류 메시지)를 먼저 읽습니다.
둘째, 어느 stage에서 멈췄는지 **Stage View**의 색깔로 확인합니다.
셋째, 오류 메시지를 그대로 검색하거나, VM 터미널에서 **같은 명령을 손으로 직접 실행**해서 원인을
재현해봅니다.

**(경험 강조)** 이 진단 과정 자체가 실무에서 개발자·엔지니어가 매일 하는 일입니다. 오늘 겪는
어려움이 곧 실력이 되는 시간이라고 생각하시면 좋겠습니다."

## [슬라이드 09] VM 실습 — E2E 테스트

"자, 직접 해보겠습니다.

`app.py`의 챗봇 응답 문구를 한 줄 수정하세요. 그다음 `git push origin main`을 실행하고, **아무것도
누르지 말고** 최대 1~2분 기다리세요.

Jenkins Build History에 자동 빌드가 새로 생기는지 확인하고, 빌드가 성공하면 `curl $(minikube
service chatbot --url)`로 실제 응답이 바뀐 걸 확인하세요."

---

## [슬라이드 10] VM 실습 — 문제 발생 시 대응

"만약 실패한다면, 실패한 stage를 확인하고 아까 본 체크리스트를 참고해서 원인부터 좁혀나가세요.

`docker ps`, `kubectl get pods`, `kubectl describe pod`처럼 실제 상태를 직접 확인하는 명령어들을
적극적으로 활용하세요.

해결되면 다시 코드를 수정해서 `git push`로 재검증하시면 됩니다. 이 반복 과정 자체가 실전
트러블슈팅입니다.

**(순회 지도 안내)** 제가 돌아다니면서 막힌 분들을 도와드리겠습니다. 편하게 손 들어주세요."
