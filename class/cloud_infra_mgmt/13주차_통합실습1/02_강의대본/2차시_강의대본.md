## [슬라이드 05] 왜 kubectl을 바로 실행하면 안 될까

"직관적으로 생각하면, Jenkinsfile의 Deploy 단계에 그냥 `kubectl set image ...`를 넣으면 될 것
같죠? 그런데 실제로 해보면 안 됩니다.

**(원인 설명)** Jenkins는 Docker 컨테이너로 떠 있고, minikube도 별도의 Docker 컨테이너로 떠
있습니다. Jenkinsfile에서 kubectl을 곧바로 실행하면, 네트워크상 minikube 클러스터에 접속이 안
됩니다. 별도 Docker 네트워크와 IP 충돌 문제 때문인데, 이건 이미 검증된 결과입니다.

**(해결 방법 제시)** 그래서 우리는 다른 경로를 씁니다 — 11주차에 Jenkins를 설치할 때 마운트했던
`docker.sock`을 이용해서, minikube 컨테이너 **'안'**에 이미 들어있는 kubectl을 그대로 실행하는
방식이에요."

## [슬라이드 07] docker exec 우회 방식

"화면의 그림을 보시면, Jenkins가 minikube에 직접 접속하는 대신, `docker exec`로 minikube 컨테이너
**안에서** 명령을 실행합니다.

**(핵심 이해)** minikube 컨테이너 내부에는 이미 kubectl과 클러스터 인증 정보(`admin.conf`)가 전부
들어있습니다. 이 경로를 그대로 활용하면, 네트워크로 직접 연결할 필요 없이도 배포 명령을 실행할 수
있는 거예요.

**(11주차 연결)** 기억나시나요? 11주차에 `docker.sock`을 마운트하는 이유를 배우면서 '13주차에
이걸 쓸 거예요'라고 예고했었죠. 바로 지금이 그 순간입니다."

## [슬라이드 09] Build & Push 단계 (12주차 문법 재사용)

"Build와 Push 단계는 12주차에 배운 문법과 거의 똑같습니다.

```groovy
stage("Build") {
  steps { sh "docker build -t {id}/chatbot:${BUILD_NUMBER} ." }
}
stage("Push") {
  steps {
    withCredentials([usernamePassword(credentialsId:"dockerhub-cred",
      usernameVariable:"USER", passwordVariable:"PASS")]) {
      sh "echo $PASS | docker login -u $USER --password-stdin"
      sh "docker push {id}/chatbot:${BUILD_NUMBER}"
    }
  }
}
```

`${BUILD_NUMBER}`로 이미지에 버전 태그를 붙이고, `withCredentials`로 Docker Hub 비밀번호를
안전하게 사용합니다."

---

## [슬라이드 10] Deploy 단계 (docker exec 방식, 검증 완료)

"이제 오늘의 핵심, Deploy 단계입니다.

```groovy
stage("Deploy") {
  steps {
    sh 'docker exec minikube $(docker exec minikube sh -c "ls /var/lib/minikube/binaries/*/kubectl") \
        --kubeconfig=/etc/kubernetes/admin.conf \
        set image deployment/chatbot web={id}/chatbot:${BUILD_NUMBER}'
    sh 'docker exec minikube ... rollout status deployment/chatbot --timeout=120s'
  }
}
```

**(한 줄씩 설명)** 안쪽의 `docker exec minikube sh -c \"ls ...\"`가 먼저 실행되어서 minikube 컨테이너
안의 kubectl 실제 경로를 찾아냅니다. 그 경로로 `--kubeconfig=/etc/kubernetes/admin.conf` 옵션을
주고 `set image` 명령을 실행해서, Deployment의 이미지를 방금 Push한 새 버전으로 교체합니다.
`rollout status`는 이 교체(Rolling Update)가 끝날 때까지 기다리는 명령입니다.

**(복잡함 인정 + 안심시키기)** 명령어가 길고 복잡해 보이죠? 템플릿 코드를 그대로 복사해서 `{id}`
부분만 본인 Docker Hub 아이디로 바꾸면 되니, 너무 부담 갖지 않으셔도 됩니다."

## [슬라이드 11] Credential 등록 먼저 하기

"오늘 실습을 시작하기 전에 꼭 먼저 해야 할 게 있습니다. Jenkins 관리 화면의 **Credentials**
메뉴에서, Docker Hub 아이디와 비밀번호를 `dockerhub-cred`라는 ID로 등록해두세요.

**(12주차 연결 재확인)** 12주차에 배운 `withCredentials`를 쓰는 이유가 바로 이겁니다 — Jenkinsfile
코드 안에는 비밀번호가 절대 그대로 적히지 않고, Jenkins가 안전하게 저장해둔 값을 실행 시점에만
주입해줍니다."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "docker exec 명령이 너무 길고 어려워요" | 이 명령어는 그대로 복사해서 `{id}` 부분만 바꾸면 되는 '템플릿'입니다. 원리(11주차 docker.sock + minikube 내부 kubectl)만 이해하면 되고, 명령어 자체를 암기할 필요는 없다고 안심시켜 주세요. |
| "kubectl 경로가 버전마다 다른가요?" | 네, `/var/lib/minikube/binaries/{버전}/kubectl`처럼 minikube 버전에 따라 경로의 일부가 달라질 수 있습니다. 그래서 `$(docker exec minikube sh -c \"ls .../*/kubectl\")`처럼 와일드카드로 경로를 자동으로 찾아내는 방식을 씁니다. |
| "Deploy 단계에서 오류가 나요" | 대부분 세 가지 원인입니다 — ① Deployment 이름(`chatbot`)이나 컨테이너 이름(`web`)이 실제 deployment.yaml과 다름, ② Docker Hub에 이미지가 실제로 Push됐는지, ③ minikube가 실행 중인지(`minikube status`) 순서대로 확인하도록 안내하세요. |
