## [슬라이드 05] 지금까지와 오늘의 차이

"지금까지는 `docker run nginx`처럼, 이미 완성된 이미지를 가져다 썼습니다. 그런데 여러분이 직접 만든
앱은 세상에 그런 이미지가 없겠죠?

Dockerfile은 **'이런 순서로 이미지를 만들어라'라고 적어두는 설계도, 일종의 레시피**입니다. 이 레시피를
`docker build` 명령으로 실제 이미지로 바꾸는 거예요.

**(기대감 조성)** 오늘부터는 여러분만의 앱을 담은, 세상에 하나뿐인 이미지를 직접 만들게 됩니다."

## [슬라이드 06] 주요 명령어 한눈에 보기

"Dockerfile에서 자주 쓰는 명령어 여섯 개를 정리했습니다.

`FROM`은 어떤 이미지를 기반으로 시작할지 정합니다. 예를 들어 파이썬 앱이면 `python:3.11-slim`처럼요.
`WORKDIR`는 컨테이너 안에서 작업할 폴더를 지정하고, `COPY`는 내 컴퓨터의 파일을 이미지 안으로 복사합니다.
`RUN`은 이미지를 만드는 동안 한 번 실행할 명령어(예: 패키지 설치)이고, `EXPOSE`는 이 컨테이너가 어떤
포트를 쓰는지 알려주는 문서 역할이에요. 마지막 `CMD`는 컨테이너가 시작될 때 실행할 명령어입니다."

## [슬라이드 08] Dockerfile이 이미지가 되기까지

"`docker build -t {이름}:{태그} .` 명령을 실행하면, Docker는 Dockerfile을 한 줄씩 순서대로 실행하면서
**레이어**를 하나씩 쌓습니다.

화면의 그림을 보시면 FROM으로 기반 이미지를 가져오고, WORKDIR로 폴더를 정하고, COPY로 파일을 넣고,
RUN으로 패키지를 설치하고, 마지막 CMD까지 오면 완성된 이미지가 됩니다. 이 레이어 개념은 3차시에서
'이미지 최적화'를 배울 때 다시 나오니 지금은 '순서대로 쌓인다'는 것만 기억해두세요."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "FROM에 아무 이미지나 써도 되나요?" | 원칙적으로는 가능하지만, 내가 만들 앱과 맞는 기반 이미지를 골라야 합니다(파이썬 앱이면 python 이미지, Node.js 앱이면 node 이미지). Docker Hub에서 공식 이미지를 검색해서 고르는 게 일반적입니다. |
| "COPY와 RUN 순서가 중요한가요?" | 네, 다음 3차시에서 자세히 다루지만 미리 짧게 답하면 — 자주 바뀌는 파일(코드)은 나중에, 잘 안 바뀌는 것(패키지 설치)은 먼저 두면 빌드 속도가 빨라집니다. |
| "GitHub 계정도 꼭 있어야 하나요?" | 네, 3주차부터는 코드를 GitHub에 올리는 것도 실습에 포함됩니다. 아직 없으시면 github.com에서 무료로 가입 가능합니다. |
