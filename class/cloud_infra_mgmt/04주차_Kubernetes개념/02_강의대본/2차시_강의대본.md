## [슬라이드 05] Pod = 컨테이너를 감싸는 껍질

"K8s는 컨테이너를 '직접' 다루지 않습니다. 대신 **Pod**라는 단위로 한 번 감싸서 다뤄요.

Pod 안에는 보통 컨테이너가 1개 들어가지만, 필요하면 여러 개를 넣을 수도 있습니다. 그리고 같은 Pod
안에 있는 컨테이너들은 네트워크(IP 주소)와 저장 공간을 서로 공유합니다.

**(질문 유도)** 왜 굳이 한 겹을 더 씌울까요? 답은 — 항상 함께 떠야 하는 컨테이너들, 예를 들어 '앱
컨테이너 + 그 앱의 로그를 수집하는 컨테이너'처럼 짝을 이루는 것들을 하나의 단위로 묶어서 관리하기
위해서입니다."

---

## [슬라이드 06] 컨테이너 vs Pod

"표로 비교하면, `docker run`으로 만든 컨테이너는 우리가 직접 관리해야 했죠. 실패하면 우리가 직접
다시 실행해야 하고요.

반면 K8s의 Pod는 K8s가 자동으로 상태를 유지해줍니다. Pod가 죽으면 — 정확히는 다음 시간에 배울
Deployment가 관리하는 Pod라면 — K8s가 알아서 새로 만들어줍니다. 배포 단위도 컨테이너 1개가 아니라
'컨테이너 1개 이상을 묶은 Pod' 단위로 바뀝니다."

## [슬라이드 08] Pod 기본 명령어

"다섯 개만 기억하시면 됩니다.

`kubectl run {이름} --image={이미지}`로 Pod 하나를 즉석에서 실행합니다. `kubectl get pods`로 지금
떠 있는 Pod 목록을 확인하고, `kubectl describe pod {이름}`으로 그 Pod의 상세 상태와 무슨 일이
있었는지(이벤트)를 확인합니다. `kubectl logs {이름}`으로 로그를 보고, `kubectl delete pod {이름}`으로
지울 수 있습니다.

**(Docker와 비교)** 눈치채셨겠지만, Docker에서 배운 run/ps/logs/rm과 거의 비슷한 구조예요. 새로
배우는 게 아니라 익숙한 패턴이 K8s에도 그대로 적용되는 겁니다."

## [슬라이드 09] Killercoda — Pod 선언 실행

"자, 직접 해보겠습니다. 지난 시간처럼 미리 구동된 멀티노드 클러스터에 접속합니다.

`kubectl run mypod --image=nginx`로 Pod를 하나 실행하고, `kubectl get pods`로 Running 상태인지
확인하세요. 그다음 `kubectl describe pod mypod`로 이 Pod가 어느 노드에 배치됐는지 확인해보세요.

**(화면 공유해 실제로 시연)** 제가 먼저 보여드릴게요."

## [슬라이드 10] VM 과제 — minikube에서 동일 실습

"개인 VM에서 `minikube start`(지난 시간에 켜두셨다면 생략 가능)로 클러스터를 켜고, 같은 명령어를

`kubectl run mypod --image=nginx`, `kubectl get pods`, `kubectl describe pod mypod`,

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "Pod 하나에 컨테이너 여러 개를 넣는 경우가 실제로 있나요?" | 네, 예를 들어 앱 컨테이너와 그 앱의 트래픽을 감시하는 사이드카 컨테이너를 함께 넣는 패턴이 실무에서 자주 쓰입니다. 이 과목에서는 다루지 않지만 존재한다는 것만 알아두면 됩니다. |
| "kubectl run으로 만든 Pod가 죽으면 자동으로 살아나나요?" | 오늘 방식(kubectl run)으로 만든 단독 Pod는 죽으면 그걸로 끝입니다. 자동으로 다시 살아나게 하려면 다음 시간에 배울 Deployment로 관리해야 합니다. |
| "describe 결과가 너무 길어요" | 정상입니다. 지금은 Events 섹션(무슨 일이 있었는지)과 Status(현재 상태) 정도만 눈여겨보면 충분합니다. |
