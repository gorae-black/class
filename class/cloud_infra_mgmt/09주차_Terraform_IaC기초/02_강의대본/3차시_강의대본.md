## [슬라이드 05] 만들 때도 코드, 지울 때도 코드

"2주차에서 컨테이너를 지울 땐 `docker rm`을 손으로 실행했었죠. Terraform으로 만든 리소스는 조금
다릅니다.

`terraform destroy` 한 번이면, 코드에 선언된 리소스들을 **전부** 제거합니다. 여러 개를 만들었어도
하나하나 지울 필요 없이 한 번에 정리되는 거예요.

**(안전장치 설명)** destroy도 apply처럼, 먼저 '무엇을 지울 예정인지' 보여주고 확인을 거칩니다.
`-auto-approve` 옵션을 붙이면 이 확인 과정을 생략할 수 있습니다.

오늘은 2차시에 만든 `tf-nginx` 컨테이너를 이 명령으로 지워보겠습니다."

## [슬라이드 07] Terraform이 상태를 기억하는 방법

"`terraform apply`를 실행하면, `terraform.tfstate`라는 파일이 자동으로 생성됩니다.

**(핵심 설명)** 이 파일 안에 '내가 지금 무엇을 만들었는지'가 기록됩니다 — 리소스의 ID, 속성값
등이 전부 담기죠.

**(그림 설명)** 화면의 그림을 보시면, HCL 코드가 '원하는 상태'를 나타내고, `terraform.tfstate`가
'현재 상태'를 기록합니다. 다음에 plan이나 apply를 실행할 때, Terraform은 이 State와 코드를
비교해서 무엇이 바뀌었는지 판단하고, 그 차이만 반영합니다."

---

## [슬라이드 08] State가 왜 필요한가

"State가 없으면 Terraform은 실제 인프라에 무엇이 있는지 전혀 알 방법이 없습니다.

**(핵심 동작 원리)** State, 실제 인프라, 코드 — 이 세 가지를 비교해서 '차이(diff)'만 반영하는
것이 Terraform의 핵심 동작 원리입니다. 예를 들어 코드에서 컨테이너 이미지 버전을 하나만 바꾸면,
Terraform은 State를 보고 '이 리소스만 업데이트하면 되는구나'라고 판단합니다. 나머지 리소스는
건드리지 않아요.

**(주의사항)** 그래서 State 파일을 사람이 직접 손으로 수정하는 건 매우 위험합니다. 반드시
`terraform` 명령을 통해서만 다뤄야 합니다."

## [슬라이드 09] 로컬 State의 한계 (10주 예고)

"그런데 지금까지 우리가 쓴 State 파일은 전부 내 VM의 로컬 디스크에만 저장되어 있었습니다. 여기엔
몇 가지 문제가 있어요.

**(문제 1)** 여러 사람이 같이 작업한다면? 각자의 State가 서로 다른 정보를 갖게 되어 충돌이 날 수
있습니다.

**(문제 2)** State 파일 안에는 비밀번호 같은 민감한 정보가 **평문으로** 포함될 수 있어서, 보안
위험도 있습니다.

**(다음 주 예고)** 이 문제를 해결하는 방법이 '원격 State'입니다. 다음 10주차에 Terraform Cloud라는
서비스를 이용해서, State를 안전하게 원격으로 관리하는 법을 직접 배워보겠습니다."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "State 파일을 지우면 어떻게 되나요?" | Terraform이 '내가 뭘 만들었는지' 기억을 잃어버립니다. 실제 인프라(컨테이너 등)는 그대로 남아있지만, Terraform은 그걸 모르는 상태가 되어 다시 apply하면 중복으로 리소스를 만들려고 시도할 수 있습니다. 그래서 State 파일은 절대 함부로 지우면 안 됩니다. |
| "destroy 없이 그냥 docker rm으로 지우면 안 되나요?" | 가능은 하지만 권장하지 않습니다. `docker rm`으로 직접 지우면 State 파일에는 여전히 그 리소스가 '존재한다'고 기록되어 있어서, State와 실제 인프라가 어긋나게(drift) 됩니다. Terraform으로 만든 건 Terraform으로 지우는 게 원칙입니다. |
| "State 파일을 여러 사람이 같이 보려면 어떻게 하나요?" | 바로 그 문제를 해결하는 게 다음 주에 배울 원격 State입니다. Terraform Cloud 같은 서비스에 State를 저장해두면, 여러 사람이 같은 State를 공유하며 안전하게 협업할 수 있습니다. |
| "Killercoda 링크가 언제 만들어졌는지에 따라 제한 시간이 달라지나요?" | 아니요. 60분 타이머는 링크 생성 시점이 아니라 각 학생이 시나리오를 '시작'한 순간부터 개별적으로 시작됩니다. 링크 자체는 만료되지 않으므로, 비동기로 늦게 접속하거나 여러 학생이 같은 링크를 써도 각자 독립된 환경과 타이머를 받습니다. |
