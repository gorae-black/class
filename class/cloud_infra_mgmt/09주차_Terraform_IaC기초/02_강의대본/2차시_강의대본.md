## [슬라이드 05] apt 저장소 등록 후 설치

"Terraform은 HashiCorp라는 회사가 공식으로 apt 저장소를 제공합니다. 순서는 이렇습니다.

먼저 HashiCorp의 공식 GPG 키를 등록하고, 그다음 apt 저장소 목록에 HashiCorp 저장소를 추가한 뒤,
`apt-get install`로 설치합니다.

화면에 나온 명령어를 그대로 실행하시면 됩니다.

```
wget -O- https://apt.releases.hashicorp.com/gpg | \
  sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg
sudo apt-get update && sudo apt-get install -y terraform
```

설치가 끝나면 `terraform -version`으로 정상 설치됐는지 확인합니다.

**(안심시키기)** 이 명령어들은 이미 여러 번 검증된 명령어이니 그대로 따라 치시면 문제없이
설치됩니다."

## [슬라이드 07] HCL 코드의 두 가지 핵심 블록

"HCL 코드는 크게 두 가지 블록으로 구성됩니다.

**provider** 블록 — 어떤 인프라를 다룰지 선언합니다. 오늘은 `provider \"docker\" {}`처럼 Docker를
다루겠다고 선언합니다.

**resource** 블록 — 실제로 만들고 싶은 대상을 선언합니다. `resource \"타입\" \"이름\" { 속성들 }`
형태로 씁니다.

**(구조 강조)** 이 두 블록의 관계를 정리하면 — provider가 '어떤 세계에서 놀 것인가'를 정하고,
resource가 '그 세계에서 무엇을 만들 것인가'를 정하는 겁니다."

---

## [슬라이드 08] main.tf 예시 — Docker 컨테이너 선언

"실제 예시를 보겠습니다.

```hcl
resource "docker_container" "web" {
  name  = "tf-nginx"
  image = "nginx:latest"
  ports { internal = 80; external = 8080 }
}
```

이 코드를 하나씩 읽어볼게요. `docker_container`가 리소스 타입, `web`이 이 리소스를 Terraform
안에서 부르는 이름입니다(실제 컨테이너 이름과는 별개예요). 그 안에 `name`(실제 컨테이너 이름),
`image`(사용할 이미지), `ports`(포트 매핑)를 지정합니다.

**(3주차 연결)** `image = \"nginx:latest\"` 이 부분, 3주차에서 `docker run` 할 때 쓰던 이미지
이름과 똑같은 개념입니다. 다만 이번엔 명령어가 아니라 코드로 선언한다는 게 다르죠."

---

## [슬라이드 09] init → plan → apply 실전 흐름

"이제 이 코드를 실제로 실행하는 흐름입니다.

`terraform init` — main.tf에 `docker` Provider가 필요하다고 선언되어 있으니, 이걸 다운로드합니다.

`terraform plan` — \"docker_container.web을 새로 만들 예정입니다\"라는 견적을 보여줍니다. 이
단계에서는 아직 아무것도 실행되지 않습니다.

`terraform apply -auto-approve` — 실제로 컨테이너를 생성합니다. `-auto-approve`는 '진행할까요?'라는
확인 질문을 건너뛰는 옵션입니다.

마지막으로 `docker ps`를 쳐서 `tf-nginx`라는 이름의 컨테이너가 떠 있는지 확인하면 끝입니다."

## [슬라이드 10] Killercoda — HCL 문법 빈칸 채우기

"자, 직접 해보겠습니다. 뼈대만 있는 main.tf 파일에서 provider와 resource 블록의 빈칸을 채워서
완성하세요.

완성했으면 `terraform init` → `terraform plan` → `terraform apply` 순서로 실행해서 결과를
확인하세요.

**(이해도 확인)** 완성한 코드가 어떤 리소스를 만드는지 스스로 한 번 설명해보세요. 단순히 빈칸을
채우는 게 아니라, 그 코드가 무슨 의미인지 이해하는 게 목표입니다.

**(화면 공유해 실제로 시연)** 제가 먼저 보여드릴게요."

## [슬라이드 11] VM 안내 — 설치 + Docker Provider 배포

"개인 VM에서 오늘 배운 전체 과정을 그대로 진행합니다.

먼저 Terraform 설치 명령어를 그대로 실행하세요(`04_템플릿코드`에 설치 스크립트를 올려두었습니다).
그다음 main.tf를 작성하고(`docker_container` 리소스), `terraform init` → `terraform plan` →
`terraform apply -auto-approve`를 실행하세요.

`docker ps`로 `tf-nginx` 컨테이너가 떠 있는 화면을 캡처해서 제출하시면 됩니다.

**(중요 — 다음 차시 예고)** 그리고 한 가지 꼭 안내해드릴 게 있는데요, 지금 만든 컨테이너를 지우지
마세요. 다음 3차시에 이 컨테이너를 `terraform destroy`로 지우는 실습을 이어서 할 겁니다."

## [슬라이드 12] 오늘의 정리

"오늘 배운 내용, 세 가지로 정리하겠습니다.

첫째, **Terraform은 HashiCorp 공식 apt 저장소로 설치한다.**
둘째, **HCL은 provider(무엇을 다룰지)와 resource(무엇을 만들지) 블록으로 인프라를 선언한다.**
셋째, **init → plan → apply 순서로 코드를 실제 인프라로 만든다.**"

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "apt-get install 중 오류가 나요" | 대부분 GPG 키 등록 경로나 저장소 목록 파일 경로 오타입니다. syllabus의 명령어를 한 글자도 빠짐없이 그대로 입력했는지 확인하고, `sudo apt-get update`를 다시 실행해보라고 안내하세요. |
| "docker run과 뭐가 다른가요? 결과는 똑같은 nginx 컨테이너인데" | 결과물(컨테이너)은 같지만 과정이 다릅니다. `docker run`은 명령어를 실행하는 순간만 기록되고 재현하려면 같은 명령어를 다시 쳐야 하지만, Terraform은 코드 파일 자체가 '원하는 상태'로 남아있어서 언제든 같은 결과를 재현하거나, 코드를 고쳐서 변경 사항을 관리할 수 있습니다. |
| "resource 이름(\"web\")과 실제 컨테이너 이름(\"tf-nginx\")이 왜 달라요?" | \"web\"은 Terraform 코드 안에서 이 리소스를 가리키는 식별자이고, `name = \"tf-nginx\"`가 실제로 Docker에 생성되는 컨테이너 이름입니다. 코드 안의 이름과 실제 인프라의 이름은 별개라는 걸 강조해주세요. |
| "Killercoda 링크가 언제 만들어졌는지에 따라 제한 시간이 달라지나요?" | 아니요. 60분 타이머는 링크 생성 시점이 아니라 각 학생이 시나리오를 '시작'한 순간부터 개별적으로 시작됩니다. 링크 자체는 만료되지 않으므로, 비동기로 늦게 접속하거나 여러 학생이 같은 링크를 써도 각자 독립된 환경과 타이머를 받습니다. |
