## [슬라이드 05] 지금까지 우리는 손으로 인프라를 만들었다

"1주차부터 지금까지, `docker run`이든 `kubectl apply`든 전부 터미널에 명령어를 **손으로** 입력해서
인프라를 만들었습니다.

**(문제 제기)** 그런데 서버가 1대일 땐 괜찮지만, 만약 10대, 100대를 똑같이 구성해야 한다면
어떨까요? 같은 명령어를 몇 번이나 반복해서 입력해야 할까요? 게다가 사람이 실행하다 보면 실수나
누락이 생기고, 나중에 '이 인프라를 어떻게 만들었더라?' 기록도 잘 안 남습니다.

**(해결책 제시)** 이 문제를 해결하는 방법이 **IaC**, Infrastructure as Code입니다. 인프라를 코드로
선언하고, 그 코드를 실행해서 생성·변경·삭제하는 방식이에요."

---

## [슬라이드 06] IaC가 주는 이점

"IaC가 주는 이점을 네 가지로 정리하면 이렇습니다.

**재현성** — 같은 코드를 실행하면 언제 어디서든 항상 같은 인프라가 만들어집니다.
**버전 관리** — 인프라 변경 이력을 Git처럼 추적할 수 있습니다.
**자동화** — 대량의 인프라도 코드 한 번 실행으로 구성할 수 있습니다.
**문서화** — 코드 자체가 '지금 인프라가 어떻게 구성되어 있는지'를 보여주는 살아있는 기록이 됩니다."

## [슬라이드 08] HCL 코드에서 실제 인프라까지

"Terraform의 동작 흐름은 이렇습니다.

먼저 **HCL**(HashiCorp Configuration Language)이라는 Terraform 전용 문법으로 '내가 원하는 인프라
상태'를 코드로 작성합니다. 이 코드를 Terraform CLI가 읽고, **Provider**를 통해 실제 인프라의 API를
호출합니다.

**(Provider 개념 강조)** 여기서 Provider가 핵심입니다. Provider는 Terraform과 각 인프라(Docker,
AWS, GCP 등) 사이를 연결해주는 플러그인이에요. 같은 HCL 문법을 쓰더라도, 어떤 Provider를 쓰느냐에
따라 Docker 컨테이너를 만들 수도 있고, AWS 서버를 만들 수도 있습니다."

---

## [슬라이드 09] 이번 학기에 쓸 Provider

"이번 학기에는 두 가지 Provider를 씁니다.

9주차 오늘부터는 **docker Provider**로, 로컬 VM의 Docker 컨테이너를 코드로 배포·통제해볼 거예요.
10주차에는 **Terraform Cloud와 random Provider**로, 원격 실행과 State 관리를 배웁니다.

**(핵심 강조)** 같은 HCL 문법으로 다양한 인프라를 다룰 수 있다는 게 Terraform의 핵심 장점입니다.
오늘 Docker로 배운 문법 구조는, 나중에 여러분이 AWS나 다른 클라우드를 다루게 되더라도 거의 그대로
적용됩니다."

---

## [슬라이드 10] 핵심 명령어 4가지 (개념만 먼저)

"오늘은 아직 실습 전이니, 명령어의 '역할'만 개념으로 짚고 가겠습니다. 실제 사용법은 다음 2차시에
직접 손으로 쳐보면서 배울 거예요.

`terraform init` — Provider를 다운로드하고 작업 폴더를 초기화합니다.
`terraform plan` — 코드를 실행하면 무엇이 바뀔지 **미리 보여줍니다**(아직 실제로 적용하지 않음).
`terraform apply` — plan에서 본 내용을 실제로 실행해서 인프라를 생성·변경합니다.
`terraform destroy` — 코드로 만든 인프라를 코드로 깔끔하게 제거합니다.

**(비유로 설명)** `plan`은 '이렇게 바뀔 예정입니다'라는 견적서라고 생각하면 이해가 쉽습니다. 실제
공사(apply)를 하기 전에 견적서를 먼저 보여주는 거죠."

---

## [슬라이드 11] Killercoda — Terraform CLI 샌드박스 체험

"자, 직접 해보겠습니다. 오늘은 이미 Terraform이 설치되어 있는 Killercoda 환경에서, 로컬 파일을
만드는 간단한 리소스로 감을 잡아볼 거예요.

`terraform init` → `terraform plan` → `terraform apply` 순서로 실행하고, 실제로 파일이 생성되는
것을 확인하세요. 그다음 `terraform destroy`로 다시 삭제해보세요.

**(핵심 감각 강조)** 오늘 실습의 목표는 복잡한 문법을 외우는 게 아니라, **'코드로 만들고 코드로
지운다'**는 Terraform의 핵심 감각을 먼저 느껴보는 겁니다.

**(화면 공유해 실제로 시연)** 제가 먼저 보여드릴게요."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "Terraform이랑 Ansible 같은 도구는 뭐가 다른가요?" | Terraform은 '인프라(서버, 네트워크, 컨테이너 자체)를 만드는' 데 강점이 있고, Ansible 같은 도구는 '이미 만들어진 서버 안에 소프트웨어를 설치·설정하는' 데 강점이 있습니다. 이 과목에서는 Terraform만 다룹니다. |
| "plan 없이 바로 apply하면 안 되나요?" | 가능은 하지만 권장하지 않습니다. plan으로 변경 사항을 미리 확인하지 않으면, 의도치 않은 리소스가 삭제되거나 변경될 위험이 있습니다. 실무에서는 반드시 plan을 먼저 검토하는 습관을 들입니다. |
| "Provider는 누가 만드나요?" | HashiCorp(Terraform 개발사)나 각 클라우드 회사(AWS, GCP 등), 또는 커뮤니티가 만들어 공개 저장소에 배포합니다. `terraform init`을 실행하면 코드에 명시된 Provider를 자동으로 다운로드합니다. |
| "Killercoda 링크가 언제 만들어졌는지에 따라 제한 시간이 달라지나요?" | 아니요. 60분 타이머는 링크 생성 시점이 아니라 각 학생이 시나리오를 '시작'한 순간부터 개별적으로 시작됩니다. 링크 자체는 만료되지 않으므로, 비동기로 늦게 접속하거나 여러 학생이 같은 링크를 써도 각자 독립된 환경과 타이머를 받습니다. |
