## 1. 1주차 복습 — 클라우드 컴퓨팅과 가상화 (17분)

### 왜 클라우드가 필요했는가 (4분)

"1주차의 출발점은 '온프레미스 방식의 한계'였습니다. 학교 수강신청 서버를 예로 들었었죠 — 평소엔
거의 안 쓰다가 수강신청 기간에만 트래픽이 몰리는 서버를, 미리 최대 용량으로 사놓고 평소엔 놀리는
게 온프레미스 방식의 비효율이었습니다.

**(핵심 재확인)** 클라우드 컴퓨팅은 이 문제를, '인터넷을 통해 필요한 만큼 컴퓨팅 자원을 빌려
쓰는 방식'으로 해결합니다. 여러분이 직접 서버를 사는 게 아니라, AWS·Azure·GCP 같은 회사가 이미
갖춰놓은 자원을 필요할 때 빌려 쓰는 거예요.

클라우드의 5대 특징, 다시 한번 빠르게 짚고 갈게요.
① 온디맨드 셀프서비스 — 클릭 몇 번으로 바로 자원 생성
② 광범위한 네트워크 접근 — 인터넷만 되면 어디서든 접속
③ 자원 풀링 — 여러 사용자가 물리 자원을 공유
④ 신속한 확장성 — 트래픽에 따라 자원을 늘렸다 줄였다
⑤ 종량제 과금 — **쓴 만큼만** 지불 (전기요금처럼)

**(시험 포인트)** 여기서 자주 나오는 함정이 '종량제 과금 = 고정 요금'이라고 착각하는 겁니다.
종량제는 정확히 반대 개념이에요, 사용한 만큼만 낸다는 뜻입니다."

### 가상화 기술 (5분)

"클라우드를 실제로 가능하게 만든 기술이 가상화였습니다.

**(구조 재설명)** 물리 서버 한 대 위에 '하이퍼바이저'라는 소프트웨어가 올라가고, 그 위에 여러 개의
VM(가상머신)이 각각 독립적으로 실행됩니다. 이때 중요한 포인트는, **VM 하나하나가 전부 자기만의
완전한 운영체제(Guest OS)를 가진다**는 겁니다. 그래서 한 물리 서버 위에서 Windows VM과 Linux VM을
동시에 띄울 수 있는 거예요.

여러분이 지금 쓰고 있는 실습 VM(cloud-lab)도 여러분의 Windows PC 위에서 VMware라는 가상화
소프트웨어로 만들어진 VM이라는 걸 기억하시죠? 1주차에 배운 개념이 여러분 눈앞에서 실제로 동작하고
있는 겁니다."

### 컨테이너 vs VM, Docker 아키텍처 (6분)

"그런데 VM 방식에는 한계가 있었습니다. VM마다 완전한 OS를 통째로 가지고 있으니 무겁고, 시작하는데
시간이 걸리죠. 이 문제를 해결한 게 **컨테이너**입니다.

**(핵심 비교)** 컨테이너는 VM과 달리, 호스트와 **OS 커널을 공유**합니다. 그래서 훨씬 가볍고 빠르게
실행됩니다. VM이 '건물 하나를 통째로 새로 짓는 것'이라면, 컨테이너는 '한 건물 안에서 나눠 쓰는
사무실'에 가깝다고 비유할 수 있어요.

| 비교 기준 | VM | 컨테이너 |
|---|---|---|
| OS | 각자 완전한 Guest OS | 호스트 커널 공유 |
| 무게 | 무거움(GB 단위) | 가벼움(MB 단위) |
| 시작 속도 | 느림(분 단위) | 빠름(초 단위) |
| 격리 수준 | 완전 격리 | 프로세스 수준 격리 |

그리고 Docker 아키텍처의 3요소, **Image → Container → Registry**도 다시 짚고 갈게요. Image는
컨테이너를 만들기 위한 '설계도'(정적인 것), Container는 그 설계도로 실제로 띄운 '실행 인스턴스'
(동적인 것)입니다. Registry는 Docker Hub처럼 이 Image들을 저장하고 공유하는 저장소고요.

**(시험 포인트)** Image와 Container를 반대로 설명하는 게 가장 흔한 실수입니다. '설계도(Image) →
실행 인스턴스(Container)' 순서를 꼭 기억하세요."

### 미니 확인문제 (2분)

"자, 여기서 확인문제 하나 풀어볼게요. **'컨테이너가 VM보다 가볍고 빠른 이유를 한 문장으로
설명하면?'** (잠시 시간을 준 뒤) 정답은 — 컨테이너는 호스트와 OS 커널을 공유해서, VM처럼 각자
완전한 OS를 따로 띄울 필요가 없기 때문입니다."

## 2. 2주차 복습 — 컨테이너 생명주기와 데이터 보존 (17분)

### 컨테이너 명령어 총정리 (5분)

"2주차에서는 컨테이너를 직접 다루는 명령어들을 배웠죠. 컨테이너의 생명주기를 따라가면서 다시
정리해볼게요.

`docker run` — 이미지로 컨테이너를 생성하고 실행
`docker ps` — 실행 중인 컨테이너 목록 확인 (`-a` 옵션을 붙이면 종료된 것까지 전부)
`docker logs {이름}` — 컨테이너가 출력한 로그 확인
`docker exec -it {이름} bash` — 실행 중인 컨테이너 내부에 직접 들어가기
`docker stop {이름}` — 컨테이너 정지
`docker rm {이름}` — 정지된 컨테이너 삭제

**(연결 설명)** 이 명령어들의 순서를 컨테이너의 일생으로 보면, run(태어남) → ps/logs/exec(살아있는
동안 관찰·조작) → stop(멈춤) → rm(삭제)의 흐름입니다."

### 컨테이너는 상태를 가진다 — 데이터 소실 문제 (4분)

"2주차의 핵심 문제의식은 '컨테이너를 삭제하면 그 안의 데이터도 함께 사라진다'는 것이었습니다.

**(왜 문제인가)** 예를 들어 웹서버 컨테이너 안에 로그 파일이나 사용자가 올린 이미지 파일이
쌓였는데, 그 컨테이너를 지우는 순간 전부 날아간다면 실무에서는 절대 쓸 수 없는 방식이겠죠. 그래서
등장한 게 컨테이너 밖에 데이터를 보존하는 방법, **Bind Mount**와 **Named Volume**입니다."

### Bind Mount와 Named Volume (6분)

"**Bind Mount**는 호스트(VM)의 특정 디렉토리를 컨테이너 안 경로에 그대로 연결하는 방식입니다.
`docker run -v 호스트경로:컨테이너경로` 형태로 씁니다. 호스트에서 파일을 수정하면 컨테이너 안에서도
즉시 반영되고, 반대도 마찬가지죠. 개발 중에 코드를 실시간으로 반영하고 싶을 때 유용합니다.

**Named Volume**은 Docker가 직접 관리하는 전용 저장공간을 연결하는 방식입니다.
`docker run -v 볼륨이름:컨테이너경로` 형태로 쓰고요. 호스트의 정확한 경로를 몰라도 되고, Docker가
저장 위치를 알아서 관리해주기 때문에 이식성이 높습니다.

**(핵심 비교 — 시험에 자주 나옴)** 두 방식의 차이를 한 문장으로 압축하면 — **'저장 위치를 누가
관리하는가'**입니다. Bind Mount는 개발자가 호스트 경로를 직접 지정(개발용), Named Volume은 Docker가
관리(운영용, 이식성 높음)."

### 미니 확인문제 (2분)

"확인문제입니다. **'Bind Mount로 연결한 컨테이너를 삭제하면, 호스트에 저장된 데이터는 어떻게
될까요?'** (잠시 시간을 준 뒤) 정답 — 그대로 남아있습니다. Bind Mount는 호스트 디렉토리 자체를
가리키는 것이므로, 컨테이너를 지워도 호스트의 파일은 영향을 받지 않습니다."

## 3. 3주차 복습 — Dockerfile과 이미지 빌드·배포 (17분)

### Dockerfile 주요 명령어 (5분)

"3주차에서는 직접 이미지를 만드는 법, Dockerfile을 배웠습니다. 주요 명령어를 예시와 함께 다시
볼게요.

```dockerfile
FROM python:3.11-slim      # 베이스 이미지 지정
COPY requirements.txt .    # 파일을 이미지 안으로 복사
RUN pip install -r requirements.txt   # 빌드 시점에 한 번 실행
COPY . .                   # 나머지 소스 코드 복사
EXPOSE 5000                # 이 컨테이너가 쓰는 포트를 문서화
CMD ["python", "app.py"]   # 컨테이너가 시작될 때 실행
```

**(시험에 자주 나오는 혼동 포인트)** `RUN`과 `CMD`를 헷갈리는 학생이 많습니다. `RUN`은 **이미지를
빌드하는 시점**에 딱 한 번 실행되고 그 결과가 이미지에 저장됩니다. `CMD`는 이미지가 아니라
**컨테이너가 시작될 때마다** 실행되는 명령입니다."

### docker build 과정과 레이어 캐싱 (7분)

"`docker build`를 실행하면, Dockerfile을 위에서부터 한 줄씩 읽으며 각 명령을 '레이어'라는 단위로
쌓아서 이미지를 만듭니다.

**(레이어 캐싱 원리)** 여기서 중요한 최적화 원리가 나옵니다 — Docker는 이전 빌드와 비교해서, 바뀌지
않은 레이어는 다시 만들지 않고 캐시를 재사용합니다. 그런데 레이어는 **순서대로** 재사용되기 때문에,
중간에 바뀐 레이어가 하나라도 있으면 그 아래(뒤)의 레이어는 전부 다시 만들어야 합니다.

그래서 나쁜 순서와 좋은 순서를 비교해볼게요.

**나쁜 순서** — 소스 코드를 먼저 복사하고 그다음 의존성을 설치하면, 코드가 한 줄만 바뀌어도
의존성 설치(`pip install`)까지 매번 다시 실행됩니다.
```dockerfile
COPY . .
RUN pip install -r requirements.txt
```

**좋은 순서** — 자주 안 바뀌는 의존성 목록을 먼저 복사해서 설치하고, 자주 바뀌는 소스 코드는
나중에 복사합니다. 코드만 바뀌면 의존성 설치 레이어는 캐시를 그대로 재사용해서 빌드가 훨씬
빨라집니다.
```dockerfile
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
```

**(핵심 요약)** 자주 안 바뀌는 걸 위에, 자주 바뀌는 걸 아래에 두는 게 레이어 캐싱 최적화의
핵심입니다."

### .dockerignore와 Docker Hub 배포 (3분)

"`.dockerignore` 파일은 이미지에 포함시키지 않을 파일을 지정합니다. `.git`, `node_modules`처럼
불필요하게 크거나 민감한 파일을 제외해서 이미지 크기를 줄이고 빌드 속도도 높입니다.

그리고 완성한 이미지를 공유하는 흐름 — `docker login` → `docker build -t {아이디}/{이미지명}:{태그}`
→ `docker push {아이디}/{이미지명}:{태그}`. 이렇게 올리면 다른 사람이 `docker pull`로 받아서 바로
쓸 수 있습니다."

### 미니 확인문제 (2분)

"확인문제입니다. **'Dockerfile에서 requirements.txt만 먼저 복사해서 설치하고, 소스 코드를 나중에
복사하는 이유는?'** (잠시 시간을 준 뒤) 정답 — 소스 코드는 자주 바뀌지만 의존성 목록은 잘 안
바뀌므로, 이 순서로 두면 코드만 수정했을 때 의존성 설치 레이어의 캐시를 재사용해서 빌드 속도가
빨라지기 때문입니다."

지키는 법(2주차) → 그 컨테이너를 직접 만들고 공유하는 법(3주차)**, 이렇게 이어집니다.

**(질문 시간)** 지금까지 내용 중에 헷갈리는 부분 있으면 질문해주세요.

2차시에서는 이 컨테이너들을 여러 개, 자동으로 관리해주는 Kubernetes를 4주차부터 7주차까지
복습하겠습니다."

---

# 2차시 — 4~7주 복습 (60분)

## 진행 순서

| 구간 | 시간 | 내용 |
|---|---|---|
| 도입 | 3분 | 1차시 연결 + K8s 전체 그림 |
| 4주차 복습 — K8s 아키텍처·Pod/Deployment | 13분 | 개념 재설명 + 미니 확인문제 |
| 5주차 복습 — Service와 내부 통신 | 13분 | 개념 재설명 + 미니 확인문제 |
| 6주차 복습 — ConfigMap/Secret·PV/PVC | 13분 | 개념 재설명 + 미니 확인문제 |
| 7주차 복습 — Ingress·자가치유 | 13분 | 개념 재설명 + 미니 확인문제 |
| kubectl 명령어 총정리 + 질문 | 5분 | 명령어 빠르게 훑기 + Q&A |
| **합계** | **60분** | |

## 도입 (3분)

"2차시를 시작하겠습니다. 1차시에서는 컨테이너 하나하나를 직접 다루는 법을 배웠다면, 4주차부터는
그 컨테이너들을 **자동으로 관리해주는** 오케스트레이션 도구, Kubernetes를 배웠습니다.

**(전체 그림 먼저 제시)** 4주차부터 7주차까지의 흐름을 미리 그려보면 이렇습니다 — Pod/Deployment로
컨테이너를 배포하고(4주차) → Service로 안정적인 접속 경로를 만들고(5주차) → ConfigMap/Secret으로
설정을 분리하고 PV/PVC로 데이터를 지키고(6주차) → Ingress로 여러 서비스를 도메인으로 묶고, K8s가
스스로 장애를 복구하는 자가치유까지(7주차). 오늘은 이 순서대로 복습합니다."

---

## 1. 4주차 복습 — K8s 아키텍처와 Pod/Deployment (13분)

### Control Plane과 Node (4분)

"K8s 클러스터는 크게 두 부분으로 나뉩니다. **Control Plane**은 클러스터 전체를 관리하는 두뇌 역할 —
스케줄링, 상태 감시를 담당합니다. **Node**는 실제로 컨테이너가 실행되는 워커입니다.

여러분이 실습에서 쓴 minikube는 이 Control Plane과 Node를 하나의 VM 안에 압축해놓은 1노드 클러스터
였다는 것도 기억해두세요."

### Pod와 Deployment (7분)

"**Pod**는 K8s에서 배포하는 **최소 단위**입니다. 하나 이상의 컨테이너를 감싸는 그릇이라고 보면
됩니다.

**Deployment**는 이 Pod를 몇 개 유지할지(`replicas`)를 선언하면, K8s가 그 상태를 계속 유지해주는
상위 오브젝트입니다.

**(핵심 개념 — 선언적 관리)** 여기서 시험에 자주 나오는 핵심 개념이 '선언적 관리'입니다. 우리가
'몇 개를 유지해라'라고 **선언**만 하면, K8s가 실제 상태를 그 선언에 맞춰 **스스로** 조정합니다.
Pod가 죽으면 자동으로 새로 만들고, 너무 많으면 줄입니다. 이 원리가 나중에 7주차 '자가치유'로 이어지니
꼭 기억해두세요.

**(예시로 확인)** `kubectl apply -f deployment.yaml`로 replicas: 2인 Deployment를 배포했다고
합시다. `kubectl get pods`로 Pod 2개가 뜬 걸 확인한 후, `kubectl delete pod {이름}`으로 하나를
지우면 어떻게 될까요? 다시 `kubectl get pods`를 치면 — 신기하게도 다시 2개로 돌아와 있습니다.
이게 선언적 관리가 실제로 작동하는 모습입니다."

### 미니 확인문제 (2분)

"확인문제입니다. **'Pod와 Deployment의 차이를 한 문장으로 설명하면?'** (잠시 시간을 준 뒤) 정답 —
Pod는 컨테이너를 감싸는 배포 최소 단위이고, Deployment는 그 Pod를 몇 개 유지할지 선언하고 K8s가
그 상태를 스스로 유지·관리하는 상위 오브젝트입니다."

---

## 2. 5주차 복습 — Service와 내부 통신 (13분)

### Service가 필요한 이유 (4분)

"5주차의 출발점은 'Pod IP가 불안정하다'는 문제였습니다. Pod가 재생성될 때마다 IP가 바뀌기 때문에,
다른 Pod나 사용자가 그 IP를 직접 바라보고 통신하면 연결이 끊어져 버립니다.

**Service**는 이 문제를 해결하는 **고정된 진입점**입니다. Pod의 IP가 바뀌어도, Service의 주소는
항상 같기 때문에 안정적으로 접속할 수 있습니다."

### Service의 세 가지 타입 (5분)

"Service에는 세 가지 타입이 있었죠.

| 타입 | 접속 범위 | 용도 |
|---|---|---|
| ClusterIP(기본값) | 클러스터 내부에서만 | Pod끼리 내부 통신 |
| NodePort | 클러스터 밖에서도(노드IP:포트) | 개발·테스트 환경 외부 접속 |
| LoadBalancer | 클라우드 로드밸런서를 통해 | 운영 환경 실제 서비스 공개 |

**(시험 포인트)** 왜 우리는 실습에서 계속 NodePort를 썼을까요? LoadBalancer는 AWS·GCP 같은 실제
클라우드 사업자의 로드밸런서가 있어야 동작하는데, minikube(로컬 환경)에는 그게 없기 때문입니다.
그래서 로컬 실습에서는 NodePort로 외부 접속을 흉내낸 거예요."

### Endpoints와 라운드로빈 (3분)

"Service 뒤에 실제로 연결된 Pod IP 목록이 **Endpoints**입니다. Pod가 추가·삭제될 때마다 이 목록도
자동으로 갱신됩니다.

그리고 Service로 요청이 여러 번 들어오면, 뒤의 여러 Pod에 **순서대로 돌아가며** 분산됩니다. 이걸
**라운드로빈**이라고 부릅니다."

### 미니 확인문제 (1분)

"확인문제입니다. **'Pod IP를 직접 쓰지 않고 Service를 쓰는 이유는?'** 정답 — Pod는 재생성될 때마다
IP가 바뀌지만, Service는 고정된 주소를 제공하므로 안정적으로 접속할 수 있기 때문입니다."

---

## 3. 6주차 복습 — ConfigMap/Secret과 PV/PVC (13분)

### ConfigMap과 Secret (5분)

"6주차는 두 축으로 나뉩니다 — '설정 분리'와 '데이터 영속성'.

**ConfigMap**은 로그 레벨, API 주소처럼 민감하지 않은 설정값을 이미지 밖에 분리해 관리합니다.
`kubectl create configmap {이름} --from-literal=KEY=VALUE`로 만들고, Pod의 환경변수로 불러와
씁니다.

**Secret**은 사용법은 ConfigMap과 거의 같지만, 비밀번호·API 키 같은 민감한 정보에 씁니다.

**(시험에 가장 자주 나오는 함정)** Secret의 값은 **Base64로 인코딩**될 뿐이지, **암호화되는 게
아닙니다.** 누구나 디코딩해서 원래 값을 볼 수 있어요. 진짜 보안은 '누가 이 Secret에 접근할 수
있는가'를 통제하는 권한 관리로 확보해야 합니다. 이 함정은 OX 문제, 서술형 문제 양쪽에서 다 나올 수
있으니 꼭 기억하세요."

### PV와 PVC (6분)

"이제 데이터 영속성 축입니다. K8s의 Pod도 결국 컨테이너를 감싼 것이므로, Pod가 삭제되면 그 안의
데이터도 사라진다는 문제가 있었죠(2주차와 같은 문제의 K8s 버전).

**PV(PersistentVolume)**는 실제로 존재하는 저장공간 그 자체입니다.
**PVC(PersistentVolumeClaim)**는 '이런 저장공간이 필요하다'는 **요청서**입니다.

**(관계 재확인)** Pod는 PVC를 통해 PV에 연결됩니다. Pod는 실제 저장공간이 어디 있는지 직접 알 필요
없이, PVC에 '이런 조건의 저장공간을 달라'고 요청만 하면 됩니다.

**hostPath**는 노드(VM)의 디렉토리를 PV로 그대로 사용하는 방식으로, 우리는 로컬 실습에서 이 방식을
썼습니다.

**(실습 연결)** 실제로 MySQL Pod에 PVC를 연결하고, Pod를 삭제했다가 다시 생성해도 데이터가 그대로
유지되는 걸 확인했었죠. 이게 PV/PVC가 실제로 동작하는 증거입니다."

### 미니 확인문제 (2분)

"확인문제입니다. **'Secret에 비밀번호를 저장하면 그 값은 자동으로 암호화되나요?'** (잠시 시간을
준 뒤) 정답 — 아닙니다. Base64로 인코딩될 뿐이며, 이는 암호화가 아닙니다. 누구나 디코딩할 수
있으므로, 진짜 보안은 접근 권한 관리로 확보해야 합니다."

---

## 4. 7주차 복습 — Ingress와 자가치유 (13분)

### Ingress (6분)

"7주차는 K8s 오브젝트 학습의 마지막 주였습니다.

서비스가 여러 개일 때 NodePort만으로는 포트 번호를 여러 개 관리해야 하는 번거로움이 있었죠.
**Ingress**는 클러스터 외부 요청을 받아, 도메인이나 경로 규칙에 따라 알맞은 Service로 안내해주는
오브젝트입니다. 이걸 **L7(애플리케이션 계층) 라우팅**이라고 부릅니다.

라우팅 방식은 두 가지 — **도메인 기반**(`blog.mysite.com` → Service A) 과 **경로 기반**
(`mysite.com/ko` → 한국어 서비스)입니다.

**(실습 연결)** 로컬 실습에서는 hosts 파일에 가짜 도메인을 minikube IP에 매핑해서 도메인 라우팅을
흉내냈습니다. 실제 서비스에서는 이 부분이 공인 DNS 등록으로 대체된다는 것도 기억해두세요.

**(시험 포인트)** Ingress는 Pod에 직접 연결되지 않고, **항상 Service를 거쳐서** 연결됩니다."

### 자가치유(Self-Healing) (5분)

"마지막으로 자가치유입니다. 4주차에서 배운 '선언적 관리'를 다시 떠올려보세요. Deployment에
선언된 '원하는 상태'(replicas 개수)와 실제 상태를 K8s가 계속 비교하고, 다르면 스스로 맞춘다고
했었죠.

**Pod가 죽거나 삭제되면, K8s가 즉시 감지하고 새 Pod를 자동으로 생성**합니다. 이게 자가치유입니다.

**(6주차와의 연결)** 여기에 6주차의 PV/PVC가 합쳐지면 — Pod가 자동으로 재생성되고, 데이터도
PV/PVC 덕분에 그대로 유지됩니다. 장애가 나도 시스템이 데이터 손실 없이 스스로 복구되는 거죠. 이게
K8s를 실무에서 널리 쓰는 핵심 이유 중 하나입니다.

**(시험 포인트)** 자가치유는 **Pod 단위 복구**입니다. Deployment 없이 단순 Pod만 떠 있는 상태에서
그 Pod를 삭제하면, 자동으로 재생성되지 않습니다 — Deployment가 관리하고 있어야 선언적 관리가
작동합니다."

### 미니 확인문제 (2분)

"마지막 확인문제입니다. **'Deployment 없이 kubectl run으로 만든 단일 Pod를 삭제하면, K8s가 자동으로
다시 만들어줄까요?'** (잠시 시간을 준 뒤) 정답 — 아닙니다. 자가치유는 Deployment 같은 상위
오브젝트가 '원하는 상태'를 선언하고 있을 때만 작동합니다. 단순 Pod에는 그 선언이 없으므로 자동으로
재생성되지 않습니다."

# 부록 — 핵심 명령어·헷갈리는 포인트 모음 (빠른 확인용)

## 핵심 kubectl 명령어 모음 (서술형 대비)

| 명령어 | 용도 |
|---|---|
| `kubectl get pods` / `-o wide` | Pod 목록·상태·IP 확인 |
| `kubectl describe pod {이름}` | Pod 상세 정보·이벤트 확인(문제 진단용) |
| `kubectl logs {이름}` | 컨테이너 로그 확인 |
| `kubectl apply -f {파일}.yaml` | YAML로 선언한 리소스 생성·갱신 |
| `kubectl delete pod {이름}` | Pod 강제 삭제(자가치유 관찰용) |
| `kubectl expose deployment {이름} --type=NodePort --port=80` | Deployment를 Service로 노출 |
| `kubectl get endpoints {서비스명}` | Service에 연결된 실제 Pod IP 목록 확인 |
| `kubectl create configmap {이름} --from-literal=K=V` | ConfigMap 생성 |
| `kubectl create secret generic {이름} --from-literal=K=V` | Secret 생성 |
| `kubectl get pv` / `kubectl get pvc` | PV/PVC 상태(Bound 여부) 확인 |

## 자주 헷갈리는 포인트 (실제 강의 중 학생 질문 기반)

- 종량제 과금은 고정 요금이 아니라 **쓴 만큼만** 지불하는 방식이다.
- Image는 설계도(정적), Container는 실행 인스턴스(동적)다 — 반대로 외우지 말 것.
- `RUN`은 **빌드 시점**에 한 번, `CMD`는 **컨테이너 시작 시**마다 실행된다.
- 레이어 캐싱: 자주 안 바뀌는 명령을 위에, 자주 바뀌는 명령을 아래에 둬야 빌드가 빨라진다.
- Secret은 암호화가 아니라 **Base64 인코딩**이다 — 진짜 보안은 접근권한 관리로 확보한다.
- PV는 저장공간 자체, PVC는 그 저장공간을 향한 **요청서**다 — 둘을 혼동하지 말 것.
- `kubectl apply` 직후 바로 `kubectl get pods`를 치면 아직 Pod가 안 떠서 "No resources found"가
  나올 수 있다 — 몇 초 기다리거나 `-w`로 지켜봐야 한다.
- Ingress와 Service의 관계: Ingress는 직접 Pod에 연결되지 않고 **항상 Service를 거쳐** 연결된다.
- 자가치유는 Pod 단위 복구다 — Deployment가 없으면(단순 Pod만 있으면) 삭제 시 자동으로 재생성되지 않는다.
