## [슬라이드 05] 컨테이너 안 데이터는 왜 사라질까

"컨테이너의 파일시스템은 그 컨테이너 자신에게 속해 있습니다. 지난 시간에 배운 대로 `docker rm`은
컨테이너를 완전히 제거하는 명령어였죠. 컨테이너가 사라지면, 그 안에 저장했던 파일도 함께 사라집니다.

**(실무 연결, 강조)** 이게 실무에서는 꽤 치명적인 문제입니다. 예를 들어 웹 서버 컨테이너를 새 버전으로
재배포할 때마다 사용자들이 올린 데이터가 전부 날아간다면, 그 서비스는 아예 운영이 불가능하겠죠."

## [슬라이드 07] Bind Mount란?

"Bind Mount는 **VM(호스트)의 특정 디렉토리를 컨테이너 내부의 특정 경로에 그대로 연결**하는 기능입니다.

그림을 보시면, 왼쪽이 VM의 `~/html` 디렉토리, 오른쪽이 컨테이너 안의 `/usr/share/nginx/html` 경로인데,
이 둘이 화살표로 연결되어 있죠. 이렇게 연결하면 컨테이너 안에서 그 경로에 파일을 쓰는 것 = VM에 실제
파일이 저장되는 것과 똑같아집니다.

**(핵심)** 그래서 컨테이너를 지워도, VM에 있는 원본 데이터는 전혀 영향을 받지 않고 그대로 남습니다."

---

## [슬라이드 08] -v 옵션 문법

"문법은 이렇습니다. `docker run -v {VM 경로}:{컨테이너 경로} {이미지}`.

콜론(:)을 기준으로 왼쪽이 VM 쪽 경로, 오른쪽이 컨테이너 안의 경로입니다. 예를 들어
`-v ~/html:/usr/share/nginx/html`이라고 하면, VM의 `~/html` 폴더가 컨테이너 안의
`/usr/share/nginx/html`에 연결됩니다.

**(nginx 동작 설명)** nginx라는 웹 서버는 원래 `/usr/share/nginx/html` 안에 있는 파일들을 웹 페이지로
보여주도록 만들어져 있습니다. 그래서 이 경로를 Bind Mount로 연결하면, VM에서 HTML 파일을 수정하는 것만
으로 웹 화면을 바꿀 수 있게 되는 거예요."

---

## [슬라이드 09] Killercoda — Bind Mount 실습

"자, 직접 해보겠습니다. 네 단계입니다.

`mkdir -p ~/html`로 폴더를 만들고, `echo '<h1>Hello Cloud!</h1>' > ~/html/index.html`로 간단한 HTML
파일을 하나 만듭니다. 그다음 `docker run -d -p 8080:80 -v ~/html:/usr/share/nginx/html --name webvol nginx`로
컨테이너를 실행하는데, `-p 8080:80`은 VM의 8080번 포트를 컨테이너의 80번 포트에 연결하는 겁니다.

마지막으로 `curl localhost:8080`을 쳐서 방금 만든 'Hello Cloud!'가 출력되는지 확인하세요.

**(화면 공유해 실제로 시연)** 제가 먼저 보여드릴게요."

## [슬라이드 11] 오늘의 정리

"오늘 배운 내용, 두 가지로 정리하겠습니다.

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "-v 옵션의 왼쪽 경로가 없으면 어떻게 되나요?" | Bind Mount의 경우 VM 쪽 경로가 반드시 미리 존재해야 합니다(mkdir로 먼저 만든 이유입니다). 없으면 오류가 나거나 빈 디렉토리로 새로 생성될 수 있어 혼란스러우니, 항상 먼저 만들어두는 습관을 강조하세요. |
| "포트 8080:80은 왜 필요한가요?" | 컨테이너는 기본적으로 격리되어 있어 외부에서 바로 접속할 수 없습니다. -p로 VM의 특정 포트를 컨테이너 포트에 연결해야 curl이나 브라우저로 접속할 수 있습니다. |
| "VM 파일을 직접 지우면 컨테이너에서도 사라지나요?" | 네. Bind Mount는 양방향으로 실시간 연결되어 있어서, VM에서 지우면 컨테이너에서도 즉시 사라지고, 반대도 마찬가지입니다. |
