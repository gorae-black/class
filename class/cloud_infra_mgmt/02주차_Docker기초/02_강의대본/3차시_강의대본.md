## [슬라이드 05] Bind Mount의 진짜 힘

"지난 시간 `webvol` 컨테이너는 지금도 계속 실행 중입니다. 이제 VM의 `~/html/index.html` 파일만
새로운 내용으로 덮어쓰면 어떻게 될까요?

정답은 — **컨테이너를 재시작하지 않아도 웹 화면이 즉시 바뀝니다.**

**(원리 설명)** 왜 그럴까요? Bind Mount는 파일을 '복사'하는 게 아니라 '연결'하는 것이기 때문입니다.
VM의 파일과 컨테이너가 보는 파일이 사실은 완전히 같은 파일이에요. 그러니 VM에서 수정하는 순간
컨테이너에서도 똑같이 바뀐 걸로 보이는 겁니다."

## [슬라이드 07] Named Volume이란?

"Named Volume은 `docker volume create`로 만드는, **Docker가 직접 관리하는 저장 공간**입니다.

Bind Mount는 우리가 VM의 정확한 경로(`~/html` 같은)를 직접 지정해야 했죠. Named Volume은 그럴 필요
없이, 그냥 이름 하나로 다룹니다. `docker run -v {볼륨이름}:{컨테이너 경로}` 형태로 씁니다.

실제로 그 데이터가 VM 어디에 저장되는지는 Docker가 알아서 관리하고, 우리는 신경 쓸 필요가 없습니다."

---

## [슬라이드 08] Bind Mount vs Named Volume

"표로 비교해보겠습니다.

경로 관리: Bind Mount는 내가 직접 VM 경로를 지정하지만, Named Volume은 Docker가 알아서 관리합니다.
용도: Bind Mount는 VM의 특정 파일(예: 소스코드)을 직접 다룰 때 좋고, Named Volume은 데이터베이스처럼
'컨테이너 전용' 데이터를 저장할 때 주로 씁니다.
이식성: Named Volume은 `docker volume` 명령으로 백업하거나 다른 곳으로 옮기기가 더 쉽습니다.

**(실무 감각)** 지금 단계에서는 '둘 다 있다' 정도로 기억해두시면 됩니다. 6주차에 MySQL 데이터베이스를
다룰 때 이 Named Volume 개념이 다시 나오니, 오늘 이름 정도는 기억해두세요."

---

## [슬라이드 09] Killercoda — 실시간 반영 + Named Volume

"자, 직접 확인해보겠습니다.

`echo '<h1>Updated!</h1>' > ~/html/index.html`로 파일을 덮어쓰고, `curl localhost:8080`을 다시
쳐보세요. 컨테이너를 재시작하지 않았는데도 'Updated!'로 바뀐 게 보이시죠?

이번엔 Named Volume을 만들어보겠습니다. `docker volume create mydata`로 볼륨을 만들고,
`docker run -d -v mydata:/usr/share/nginx/html --name webvol2 nginx`로 새 컨테이너를 실행해보세요."

## [슬라이드 11] 오늘의 정리

"오늘 배운 내용, 두 가지로 정리하겠습니다.

첫째, **Bind Mount는 파일을 '복사'가 아니라 '연결'해서 실시간으로 반영된다**.
둘째, **Bind Mount는 내가 경로를 관리하고, Named Volume은 Docker가 관리한다**."
