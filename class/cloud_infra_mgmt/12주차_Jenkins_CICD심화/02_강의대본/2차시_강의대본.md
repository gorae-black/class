## [슬라이드 05] 3주차 Flask 앱을 다시 활용

"오늘은 완전히 새로운 걸 만들지 않고, **3주차에 GitHub에 올렸던 cloud-lab 저장소**(`app.py`,

**(오늘의 목표 명확화)** 오늘은 이 저장소를 Jenkins가 **Checkout**(가져오기)해서 `docker build`까지
자동으로 실행하는 Jenkinsfile을 작성할 겁니다.

**(13주차 연결)** 이건 사실 13주차 대통합 실습에서 쓸 Jenkinsfile의 축소판입니다. 오늘 Checkout과
Build만 해보면, 13주차에는 여기에 Push와 Deploy 단계만 추가하면 되는 거예요."

---

## [슬라이드 06] Checkout + Build 단계

"실제 코드를 보겠습니다.

```groovy
pipeline {
  agent any
  stages {
    stage("Checkout") {
      steps { git url: "https://github.com/{id}/cloud-lab.git", branch: "main" }
    }
    stage("Build") {
      steps { sh "docker build -t myflask:test ." }
    }
  }
}
```

`Checkout` stage는 `git url: \"...\"` 스텝으로 GitHub 저장소를 가져옵니다. `{id}` 자리에는 여러분의
GitHub 아이디를 넣어야 합니다. `Build` stage는 `sh` 스텝으로 셸 명령어(`docker build`)를 실행합니다.

**(문법 연결)** `sh` 스텝 안에 들어가는 명령어는 사실 여러분이 3주차에 터미널에서 직접 쳤던 것과
똑같은 `docker build` 명령어입니다. 다만 이제는 Jenkins가 대신 실행해주는 거예요."

## [슬라이드 09] 결과를 눈으로 확인하기

"Build Now를 실행한 뒤 확인할 게 두 가지입니다.

첫째, Stage View에서 각 단계가 **초록색**(성공)으로 표시되는지 확인합니다. 빨간색이면 그 단계에서
오류가 난 거예요.

둘째, Console Output을 열면 `git clone` 로그와 `docker build` 로그가 순서대로 출력된 걸 볼 수
있습니다. 빌드가 끝난 후 VM 터미널에서 `docker images`를 쳐보면, `myflask:test`라는 이미지가 실제로
만들어진 걸 확인할 수 있어요.

**(13주차 연결 재확인)** 오늘의 Checkout+Build가, 13주차에는 Push와 Deploy까지 이어져서 전체
파이프라인이 완성됩니다."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "3주차 저장소를 못 찾겠어요/삭제했어요" | GitHub에서 본인 계정의 저장소 목록을 확인하도록 안내하세요. 만약 삭제했다면, 3주차 자료를 참고해 `app.py`, `Dockerfile`을 다시 만들고 새 저장소에 push하면 됩니다. |
| "Checkout 단계에서 인증 오류가 나요" | 저장소가 Private이면 Jenkins가 접근할 수 없어 오류가 납니다. 저장소가 Public으로 설정되어 있는지 GitHub 저장소 설정에서 확인하도록 안내하세요. |
| "Pipeline script 대신 다른 방법은 없나요?" | 'Pipeline script from SCM'이라는 옵션으로, GitHub 저장소 안에 있는 Jenkinsfile 파일을 직접 읽어오게 할 수도 있습니다. 이 방식은 다음 3차시에 Poll SCM과 함께 다룹니다. |
