## [슬라이드 05] Webhook — 가장 흔히 쓰는 방식

"첫 번째는 **Webhook** 방식입니다. GitHub이 push를 감지하면, 즉시 Jenkins 서버로 '지금
push됐어요'라는 알림(HTTP 요청)을 직접 보냅니다.

**(장점)** 실시간에 가깝게 빠르다는 장점이 있습니다. 하지만 조건이 하나 있어요 — GitHub이 Jenkins
서버에 **직접 접속**할 수 있어야 합니다. 즉, Jenkins 서버가 인터넷에서 접근 가능한 공인 주소를
가지고 있어야 동작합니다."

---

## [슬라이드 06] 우리 VM은 왜 Webhook을 못 쓸까

"**(핵심 문제 제기)** 그런데 우리 개인 VM은 어떨까요? 학생 여러분의 VM은 가정용 공유기 NAT 뒤에
있습니다. 즉, 외부(GitHub)에서 직접 접속할 수 있는 공인 주소가 없어요.

그래서 GitHub이 'push됐다'는 알림을 우리 VM의 Jenkins로 보낼 방법이 아예 없습니다. Webhook 방식은
우리 환경에서는 애초에 동작할 수 없는 거예요.

**(해결책 제시)** 이 문제를 해결하는 방식이 오늘 배울 **Poll SCM**입니다. GitHub이 우리에게
알려주는 게 아니라, 반대로 **Jenkins가 스스로 GitHub에 물어보러 가는** 방식입니다.

**(4주차 연결)** 이런 네트워크 제약, 낯설지 않으시죠? 13주차에 Jenkins와 minikube를 연결할 때도
비슷한 네트워크 문제를 만나게 되는데, 그때도 오늘과 비슷한 발상의 전환으로 해결합니다."

## [슬라이드 08] Poll SCM 동작 원리

"Poll SCM의 동작 원리는 간단합니다.

Jenkins가 정해진 주기마다(예: 매 1분) GitHub 저장소에 '변경 사항 있어?'라고 직접 물어봅니다. 변경이
있으면 그때 자동으로 빌드를 시작하고, 없으면 아무 일도 하지 않습니다.

**(핵심 포인트)** 여기서 중요한 건, 학생 VM이 **외부에서 접근 가능할 필요가 전혀 없다**는
겁니다. Jenkins가 능동적으로 나가서 확인하는 구조라서, NAT 뒤에 있어도 아무 문제가 없어요."

---

## [슬라이드 09] Job 설정에서 Poll SCM 켜기

"실제 설정은 이렇습니다.

Job 설정 화면에서 **Build Triggers** 섹션을 찾아 **'Poll SCM'**을 체크합니다. 그리고 Schedule
입력란에 cron 형식으로 `* * * * *`를 입력합니다 — 이건 '매 1분마다 확인한다'는 뜻이에요.

저장한 뒤 아무 변경 없이 기다려보면, 빌드가 실행되지 않는 걸 확인할 수 있습니다(정상입니다 —
변경이 없으니까요). 그런데 코드를 수정하고 `git push`하면, **수 초에서 1분 이내**에 자동으로
빌드가 시작됩니다. 이건 이미 검증된 동작이니 안심하고 진행하시면 됩니다."

---

## [슬라이드 10] Killercoda — Poll SCM 설정 실습

"자, 직접 해보겠습니다. 제공된 Pipeline Job에 Poll SCM(`* * * * *`) 설정을 적용해보세요.

제가 미리 준비한 저장소에 변경 사항을 시뮬레이션할 텐데, 그 순간 자동 빌드가 트리거되는지 함께
관찰해보겠습니다. Build History를 보면 'SCM 변경 감지'로 자동 실행된 빌드가 나타나는 걸 확인할 수
있어요.

**(화면 공유해 실제로 시연)** 제가 먼저 보여드릴게요."

## [슬라이드 12] 오늘의 정리

"오늘 배운 내용, 두 가지로 정리하겠습니다.

첫째, **Webhook은 GitHub이 직접 알려주는 방식, Poll SCM은 Jenkins가 스스로 물어보는 방식이다.**
둘째, **NAT 뒤에 있는 학생 VM 환경에서는 Poll SCM(예: 매 1분 확인)으로 git push 시 자동 빌드를
구현한다.**"

|---|---|
| "매 1분마다 확인하면 서버에 부담되지 않나요?" | 개인 실습 환경에서는 큰 부담이 없습니다. 실무에서 대규모로 운영할 때는 폴링 주기를 더 길게 하거나(예: 5분), 가능하다면 Webhook 방식을 우선 고려합니다. 이 과목은 학습 목적상 빠른 피드백을 위해 1분으로 설정합니다. |

| "실제 회사에서도 Poll SCM을 쓰나요?" | 사내망이 아니라 외부에서 접근 가능한 CI 서버를 쓰는 경우에는 보통 Webhook을 씁니다. Poll SCM은 오늘처럼 네트워크 제약이 있는 환경에서 쓰는 대안적인 방식입니다. 원리를 이해해두면 어떤 환경에서든 응용할 수 있습니다. |
| "Killercoda 링크가 언제 만들어졌는지에 따라 제한 시간이 달라지나요?" | 아니요. 60분 타이머는 링크 생성 시점이 아니라 각 학생이 시나리오를 '시작'한 순간부터 개별적으로 시작됩니다. 링크 자체는 만료되지 않으므로, 비동기로 늦게 접속하거나 여러 학생이 같은 링크를 써도 각자 독립된 환경과 타이머를 받습니다. |
