## [슬라이드 05] docker run 실행 시 벌어지는 일

"`docker run nginx`라고 치는 순간, Docker 데몬은 네 단계를 거칩니다.

첫째, 내 컴퓨터에 nginx 이미지가 이미 있는지 확인합니다. 없으면 Docker Hub 같은 Registry에서 내려받아요
(pull).
둘째, 그 이미지로부터 컨테이너가 쓸 파일시스템을 준비합니다.
셋째, 격리된 실행 공간 — 리눅스의 '네임스페이스'라는 기술을 이용해서 — 을 만듭니다.
넷째, 그 격리된 공간 안에서 nginx 프로세스를 실제로 실행합니다.

**(쉬운 설명)** 네임스페이스가 어려운 용어인데, 지금은 '이 컨테이너 안에서만 보이는 방 하나를 만든다'
정도로 이해하시면 충분합니다."

---

## [슬라이드 06] 컨테이너 = '격리된' 일반 프로세스

"이게 오늘 가장 중요한 개념인데요. **컨테이너는 사실 특별한 게 아니라, 호스트 OS 위에서 돌아가는 그냥
일반 프로세스입니다.** 다만 자기만의 파일시스템·네트워크·프로세스 목록을 '보게' 격리되어 있을 뿐이에요.

그래서 컨테이너 안에 들어가서 `ps`를 치면 그 컨테이너 안의 프로세스만 보입니다. 호스트의 다른 프로세스는
안 보여요. 반대로 호스트에서 보면, 컨테이너는 그냥 프로세스 하나로 보입니다.

**(핵심 강조)** 그래서 컨테이너를 지운다는 건 VM처럼 별도의 OS를 끄는 게 아니라, 그냥 이 격리된
프로세스를 종료하는 겁니다. 이게 컨테이너가 VM보다 훨씬 빠르고 가벼운 근본적인 이유예요."

## [슬라이드 08] 오늘 실행할 명령어 미리보기

"오늘 실행할 명령어 네 개를 먼저 보여드릴게요.

`docker run -d --name web nginx` — `-d`는 백그라운드로 실행하라는 뜻이고, `--name web`은 컨테이너에
'web'이라는 이름을 붙이는 겁니다. 이름을 안 붙이면 Docker가 랜덤한 이름을 자동으로 붙여줘요.

`docker ps`로 실행 중인 컨테이너 목록을 확인하고,
`docker inspect web`으로 이 컨테이너의 모든 설정 정보를 JSON 형태로 확인합니다.
그 정보가 굉장히 많으니까, `docker inspect web | grep IPAddress`로 그중 IP 주소만 뽑아서 볼 거예요."

---

## [슬라이드 09] docker inspect가 왜 중요한가

"`inspect`가 왜 중요하냐면, 컨테이너는 기본적으로 격리되어 있어서 밖에서 내부 상태가 잘 안 보입니다.
`inspect`는 그 컨테이너의 IP 주소, 마운트된 볼륨, 환경변수, 현재 상태 같은 걸 전부 보여주는 명령어예요.

**(실무 연결)** 실무에서 컨테이너가 이상하게 동작할 때 가장 먼저 치는 명령어가 바로 이 `docker inspect`
입니다. 오늘 목표는 nginx 컨테이너를 실행하고, inspect로 그 컨테이너의 IP 주소를 직접 찾아내는 겁니다."
