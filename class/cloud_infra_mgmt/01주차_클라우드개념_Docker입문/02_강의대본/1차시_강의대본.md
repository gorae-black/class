## [슬라이드 05] 기존 온프레미스 방식의 한계

"클라우드를 이해하려면 먼저 클라우드가 나오기 전엔 어떻게 했는지를 알아야 합니다.

예전에는, 그리고 지금도 일부 회사들은, 서버를 회사가 **직접 구매**해서 자체 전산실에 설치하고 운영합니다.
이런 방식을 '온프레미스(On-premise)'라고 부릅니다. '현장에 있다'는 뜻이에요.

이 방식의 문제가 뭐냐면— 예를 들어볼게요. 우리 학교 수강신청 시스템을 생각해보세요. 수강신청 기간에는
평소보다 몇 배, 몇십 배의 트래픽이 몰립니다. 그런데 서버를 직접 운영한다면, 이 순간을 위해서 평소에는
쓰지도 않을 만큼 넉넉한 서버를 미리 사둬야 해요. 안 그러면 수강신청 당일에 서버가 다운되겠죠.

그런데 평소엔 그 비싼 서버가 놀고 있는 겁니다. 이게 바로 온프레미스 방식의 가장 큰 비효율입니다.
게다가 서버에 장애가 나면 담당자가 직접 하드웨어를 뜯어봐야 하니 대응도 느립니다."

**(질문 유도)** "여러분 중에 수강신청 때 서버 터지는 거 경험해보신 분 있으시죠? 바로 이런 이유 때문입니다."

## [슬라이드 06] 클라우드 컴퓨팅의 정의와 특징

"이 문제를 해결하기 위해 등장한 게 클라우드 컴퓨팅입니다.

클라우드 컴퓨팅을 한 문장으로 정의하면, **인터넷을 통해서 서버, 저장공간, 네트워크 같은 컴퓨팅 자원을
필요한 만큼 빌려 쓰는 방식**입니다. 여러분이 직접 서버를 사는 게 아니라, AWS나 Azure, GCP 같은 회사가
이미 갖춰놓은 자원을 필요할 때 필요한 만큼 빌려 쓰는 거죠.

특징이 다섯 가지 있는데 하나씩 빠르게 짚을게요.

하나, **온디맨드 셀프서비스** — 사람한테 전화하거나 신청서 낼 필요 없이 클릭 몇 번으로 바로 서버를 만들 수 있습니다.
둘, **광범위한 네트워크 접근** — 인터넷만 되면 어디서든 접속 가능합니다.
셋, **자원 풀링** — 여러 사용자가 물리적인 자원을 공유하면서 필요한 만큼 나눠 씁니다.
넷, **신속한 확장성** — 아까 수강신청 예시처럼 트래픽이 몰리면 순식간에 서버를 늘렸다가, 끝나면 다시 줄일 수 있어요.
다섯, **종량제 과금** — 이게 제일 체감이 클 텐데, 전기요금처럼 **쓴 만큼만** 돈을 냅니다."

## [슬라이드 09] 가상화란? — Hypervisor 기반 VM 구조

"가상화가 뭐냐면, **물리 서버 한 대를 여러 대처럼 쪼개서 쓰는 기술**입니다.

화면의 그림을 보시면, 맨 아래가 실제 하드웨어, 그 위에 '하이퍼바이저(Hypervisor)'라는 소프트웨어가 있고,
그 위에 가상머신, 즉 VM이 여러 개 올라가 있습니다. 이 하이퍼바이저가 하는 일이 CPU, 메모리, 디스크 같은
물리 자원을 나눠서 각 VM에게 배분해주는 겁니다.

중요한 건, **VM 하나하나가 전부 자기만의 완전한 운영체제(Guest OS)를 가지고 있다**는 점입니다. 그래서 한
물리 서버 위에서 Windows VM도 띄우고 Linux VM도 동시에 띄울 수 있는 거예요."

## [슬라이드 10] VM 방식의 한계

"그런데 이 VM 방식에 문제가 있습니다. 세 가지로 정리할 수 있어요.

첫째, **무겁습니다**. VM 하나마다 완전한 OS를 통째로 담고 있으니까, 이미지 용량이 수 GB에서 수십 GB까지 나갑니다.
둘째, **느립니다**. VM을 켤 때마다 그 안의 OS를 처음부터 부팅해야 하니까 수십 초에서 몇 분씩 걸려요.
셋째, **낭비됩니다**. VM을 여러 개 동시에 띄우면, 그 안의 OS들이 CPU랑 메모리를 계속 중복해서 잡아먹습니다.

**(강조)** 그래서 이런 질문이 나오게 됩니다 — '앱 하나 실행하려고 OS를 통째로 복제해야 할까?' 바로 이 질문에서
컨테이너 기술이 시작된 겁니다."

## [슬라이드 12] 컨테이너 구조 — OS 커널을 공유한다

"컨테이너는 VM과 발상 자체가 다릅니다. OS를 통째로 복제하지 않고, **호스트 OS의 커널을 여러 컨테이너가
함께 공유**합니다.

그림을 보시면 VM 그림과 거의 똑같은 구조인데, 딱 하나 다른 게 뭐냐면— 파란 박스 안에 있던 'Guest OS'가
사라지고 그냥 'App'만 남았습니다. 각 컨테이너는 앱을 실행하는 데 꼭 필요한 코드와 라이브러리만 담고 있어요.

그 결과 컨테이너는 수십 MB 정도로 작고, 시작하는 데 1초도 안 걸립니다. 아까 VM의 세 가지 단점이 전부
해결된 거죠."

---

## [슬라이드 13] VM vs 컨테이너 비교표

"이 표가 오늘 수업에서 가장 중요한 표입니다. 위에서부터 하나씩 짚어볼게요.

가상화 단위: VM은 하드웨어 전체를, 컨테이너는 OS 커널만 가상화합니다.
포함 요소: VM은 OS 전체와 앱을, 컨테이너는 앱과 필요한 라이브러리만 포함합니다.
용량: 여기 차이가 크죠 — VM은 수 GB~수십 GB, 컨테이너는 수십 MB~수백 MB.
부팅 속도: VM은 수십 초~수 분, 컨테이너는 1초 내외.
격리 수준: 여기서는 VM이 더 강합니다 — VM은 완전히 격리되지만 컨테이너는 커널을 공유하는 만큼 격리가
상대적으로 약합니다.

**(균형 잡힌 시각 강조)** 그렇다고 컨테이너가 무조건 좋은 건 아닙니다. 격리가 중요한 상황에서는 VM을 쓰기도
하고, 실무에서는 VM 위에 컨테이너를 올리는 방식도 많이 씁니다. 상황에 맞게 고르는 게 중요해요."

## [슬라이드 14] Docker 아키텍처 — Image, Container, Registry

"이제 컨테이너를 실제로 다루는 도구, Docker 이야기를 해보겠습니다. Docker를 구성하는 네 가지 요소가 있어요.

**Docker Engine** — 이미지를 내려받고, 컨테이너를 만들고 실행하고 관리하는 핵심 엔진입니다.
**Image** — 컨테이너를 실행하는 데 필요한 모든 걸 담은 '설계도'예요. 읽기 전용입니다.
**Container** — 그 설계도(Image)를 실제로 실행한 상태, 즉 '실체'입니다.
**Registry**, 대표적으로 Docker Hub — 이런 이미지들을 저장하고 다른 사람과 공유하는 저장소입니다.

**(비유로 마무리)** 이해하기 쉽게 비유하면, **Image는 붕어빵 틀, Container는 실제로 구워진 붕어빵**입니다.
같은 틀 하나로 붕어빵을 여러 개 구울 수 있죠? 마찬가지로 이미지 하나로 컨테이너를 여러 개 동시에 실행할
수 있습니다.

이따 실습에서 `docker images`를 치면 그게 여러분이 갖고 있는 '붕어빵 틀 목록'이고, `docker ps`를 치면 지금
'구워지고 있는 붕어빵 목록'이라고 생각하시면 됩니다."

받아진 이미지를 확인하고, `docker ps -a`로 실행된 컨테이너 목록을 확인할 거예요.

**(화면 공유해 실제로 시연)** 제가 먼저 화면으로 보여드릴게요. (Killercoda 열어서 `docker run hello-world`
입력 → 결과 출력 확인 → `docker images` → `docker ps -a` 순서로 라이브 시연)"

## [슬라이드 17] 로컬 VM 환경 최종 점검 (0주차 확인)

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "가상머신이랑 컨테이너 둘 다 왜 배워야 하나요?" | 실무에서는 둘 다 쓰입니다. 클라우드 인프라 자체는 VM(예: AWS EC2)으로 이루어져 있고, 그 위에서 컨테이너(Docker)를 돌리는 경우가 많습니다. 이후 주차에서 자세히 다룹니다. |
| "Killercoda가 초기화되면 지금까지 한 게 다 사라지나요?" | 네, Killercoda는 세션 기반이라 60분 후 초기화됩니다. 오늘 실습은 짧아서 문제없지만, 장기 보존이 필요한 과제는 로컬 VM에서 진행하도록 설계된 이유가 이것입니다. |
| "Docker Hub가 뭔가요? 회원가입 해야 하나요?" | Docker Hub는 이미지를 올리고 받을 수 있는 공개 저장소입니다. 오늘은 이미지를 받기만 하므로 회원가입이 필요 없습니다. 3주차부터 이미지를 올릴 때 계정이 필요합니다. |
| "제 노트북 사양이 부족한데 VM을 꼭 만들어야 하나요?" | 최소 사양(디스크 30GB, RAM 8GB)을 다시 확인해주시고, 정말 어려운 경우 조교/교수자에게 개별 상담을 요청하도록 안내하세요. |
