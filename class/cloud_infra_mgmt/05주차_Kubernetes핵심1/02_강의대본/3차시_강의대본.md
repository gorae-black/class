## [슬라이드 05] Service의 '연결 장부'

"1차시에서 Service는 라벨로 Pod를 찾는다고 배웠죠. 그 결과, 실제로 연결된 Pod들의 IP 목록이 바로
**Endpoints**입니다.

Pod가 새로 추가되거나 삭제될 때마다 이 Endpoints 목록도 자동으로 갱신됩니다. 마치 Service가 관리하는
'연결 장부' 같은 거예요.

`kubectl get endpoints {서비스이름}`을 치면 지금 이 순간 실제로 연결되어 있는 Pod IP 목록을 그대로
볼 수 있습니다."

## [슬라이드 07] 요청이 순서대로 나뉜다

"Service로 요청이 여러 번 들어오면, 뒤에 있는 Pod들에게 **순서대로 돌아가며** 전달됩니다. 이걸
'라운드로빈'이라고 해요.

그림을 보시면, Pod가 2개일 때 1번째 요청은 Pod A로, 2번째는 Pod B로, 3번째는 다시 Pod A로 — 이런
식으로 번갈아가며 전달됩니다.

**(효과 설명)** 이렇게 하면 특정 Pod 하나에만 부하가 몰리지 않고, 여러 Pod가 골고루 일을 나눠서
처리하게 됩니다."

---

## [슬라이드 08] 직접 확인하는 방법

"이걸 직접 확인하는 방법은 간단합니다. `for i in $(seq 1 10); do curl $(minikube service myapp
--url); done`처럼 curl을 10번 반복해서 요청을 보내는 거예요.

**(주의사항 안내)** 다만 오늘 쓰는 nginx 기본 페이지는 어느 Pod가 응답하든 내용이 똑같아서, curl
결과만 봐서는 어느 Pod가 응답했는지 구분이 잘 안 됩니다. 그래서 `kubectl logs`로 각 Pod의 로그를
같이 확인하면서, 요청이 두 Pod에 번갈아 기록되는 걸 보는 방식으로 확인할 거예요."

---

## [슬라이드 09] Killercoda — Endpoints·라운드로빈 확인

"자, 직접 해보겠습니다.

`kubectl get endpoints myapp`으로 연결된 Pod IP 2개를 확인하세요. 그다음 반복 curl을 10번 실행하고,
`kubectl logs {파드이름}`으로 각 Pod의 로그에 요청이 기록되는지 확인해보세요. 두 Pod의 로그를 각각
확인하면 번갈아가며 요청을 처리한 게 보일 겁니다.

**(화면 공유해 실제로 시연)** 제가 먼저 보여드릴게요."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "라운드로빈 순서를 내가 정할 수 있나요?" | 기본적으로는 순차적으로 분산되지만, 세션 유지(Session Affinity) 같은 고급 설정으로 특정 클라이언트를 항상 같은 Pod로 보낼 수도 있습니다. 이 과목에서는 다루지 않는 심화 주제입니다. |
| "Endpoints와 Pod IP가 다른가요?" | Endpoints에 나열되는 IP가 곧 그 시점의 Pod IP입니다. Endpoints는 그 IP 목록을 자동으로 최신 상태로 관리해주는 오브젝트라고 이해하면 됩니다. |
| "Pod가 3개 이상이면 라운드로빈이 더 복잡해지나요?" | 원리는 동일합니다. Pod가 몇 개든 순서대로 돌아가며 요청을 분산합니다. |
