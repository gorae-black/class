+------------------+--------------------------------------------------+
| 교과목명         | 클라우드인프라관리                               |
| 학점/시수        | 3학점 / 주 3차시 (각 1시간)                      |

※ 본 교과목은 하이브리드(Dual-Track) 전략으로 운영됩니다. 수업 중 빠른 체험은
   Killercoda(브라우저 기반)로, 시간 제약 없는 심화·과제 실습은 학생 개인 로컬
   VM 환경을 활용합니다. 13~14주차 통합 실습은 60분 타임아웃 제한이 있는
   Killercoda를 배제하고 100% 로컬 VM 전용으로 운영합니다.


--------------------------------------------------------------------------------
1. 교과목 개선 배경
--------------------------------------------------------------------------------
  * 기존 교과목은 학생 PC에 멀티노드 Kubernetes 클러스터를 직접 설치하는 방식으로
    운영되어, PC 사양·OS 충돌 등 환경적 변수로 인한 실습 장애가 빈번하였습니다.
  * Docker·K8s 중심의 단일 도구 구성으로 현장에서 요구하는 인프라 자동화(IaC)·
    배포(CI/CD) 역량을 충분히 다루기에 한계가 있었습니다.
  * 이를 개선하여 브라우저 기반 실습(Killercoda)과 로컬 VM 영구 실습 환경을
    병행하는 하이브리드 Dual-Track 전략으로 전면 재구성하였습니다.
  * Terraform(IaC)·Jenkins(CI/CD)를 커리큘럼에 추가하여 클라우드 인프라 자동화
    전 과정을 단계적으로 학습할 수 있도록 확장하였습니다.

3. [0주차] 필수 선행 단계 — 사전 구성 VM 배포
--------------------------------------------------------------------------------
  목적: 클라우드·리눅스를 처음 접하는 학생 특성을 고려하여, 학생이 직접 설치·
        구성하는 과정을 최소화한다. 교수자가 Docker·minikube·kubectl까지 전부
        설치·초기화해 둔 VM 이미지를 통째로 배포하여, 학생은 "가져오기 →
        로그인"만으로 1~7주차 실습을 바로 시작할 수 있게 한다.

  +-------------------+--------------------------------------------------------+
  | 항목              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 배포 방식         | 교수자가 사전 빌드한 VM 이미지(cloud-lab.vmx +         |
  |                   | cloud-lab.vmdk, 압축 zip)를 0주차 LMS에 탑재           |
  | 학생 활동         | ① VMware Workstation Player 설치(무료)                 |
  |                   | ② zip 압축 해제 ③ cloud-lab.vmx 열기 ④ VM 실행         |
  |                   | ⑤ student 계정으로 로그인 (별도 명령어 실행 불필요)    |
  | 계정 정보         | 아이디: student / 비밀번호: a1234 (sudo 비밀번호 없음) |
  | 목표 환경         | Ubuntu 24.04 LTS + Docker(+Compose) 사전 설치,         |
  |                   | minikube·kubectl 사전 설치 및 클러스터 초기화 완료      |
  |                   | (1~3주차 Docker, 4~7주차 Kubernetes 실습 모두 대응)    |
  | 사전 pull 이미지  | hello-world, nginx, mysql / ingress addon 이미지 캐시  |
  | VM 사양           | 2 vCPU / RAM 4GB(minikube 2.2GB로 제한 설정) / 30GB    |
  | 환경 검증         | 로그인 후 docker --version / kubectl version --client  |
  |                   | / minikube status 정상 출력 확인                        |
  | 필요 사양(호스트) | 디스크 여유 공간 15GB 이상 / RAM 8GB 이상 권장         |
  +-------------------+--------------------------------------------------------+

  # 학생 PC에서 할 일 (명령어 없이 GUI로 진행)
  1) VMware Workstation Player 설치
  2) cloud-lab-vm.zip 압축 해제
  3) VMware Player 실행 -> "Open a Virtual Machine" -> cloud-lab.vmx 선택
  4) "Play virtual machine" 클릭
  5) 로그인 프롬프트에 student / a1234 입력

  # VM 내부 검증
  docker --version
  kubectl version --client
  minikube status

4. 주차별 하이브리드 상세 실습 계획
--------------------------------------------------------------------------------

  ▣ 1단계 | Docker 컨테이너 기초 (1~3주차)

  [1주차] 클라우드 패러다임 변화와 Docker 환경 구축
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | 가상화 기술(Hypervisor vs Container) 이해 및 Docker    |
  |                   | 아키텍처 이론                                          |
  | Killercoda        | 브라우저 터미널 접속 -> docker run hello-world 실행     |
  | (수업용)          | -> 기본 레이어 구조 맛보기                              |
  | 로컬 VM           | 0주차 배포받은 개인 VM 내부에서 Docker 데몬 구동 확인,  |
  | (과제/복습)       | 독립적인 컨테이너 프로세스가 동작하는 원리 확인         |
  +-------------------+--------------------------------------------------------+

  [1주차 차시 구성]
  +--------+----------------------------------------------------------------+
  | 1차시  | 클라우드 패러다임 + 가상화(Hypervisor) 개념                     |
  | 2차시  | 컨테이너 vs VM 비교 + Docker 아키텍처(Image/Container/Registry) |
  | 3차시  | Docker 데몬 동작 원리 + nginx 컨테이너 실행·inspect 심화 실습    |
  +--------+----------------------------------------------------------------+

  # [Killercoda] 수업 실습
  docker run hello-world
  docker images
  docker ps -a

  # [로컬 VM] 과제 실습 (VM 로그인 후 바로 실행)
  docker run -d --name web nginx
  docker ps
  docker inspect web | grep IPAddress

  [2주차] Docker 컨테이너 생명주기와 데이터 보존
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | 컨테이너 명령어(run, ps, stop, rm), 볼륨 마운트         |
  |                   | (Volume/Bind Mount) 개념                                |
  | Killercoda        | Nginx 컨테이너 빠른 실행 -> 포트 포워딩으로 웹 화면      |
  | (수업용)          | 띄우기 따라하기                                         |
  | 로컬 VM           | 로컬 VM 디렉토리와 Nginx 경로를 볼륨 바인딩, 로컬 HTML  |
  | (과제/복습)       | 수정 시 웹 화면 실시간 변경 확인(데이터 영속성 원리)    |
  +-------------------+--------------------------------------------------------+

  [2주차 차시 구성]
  +--------+----------------------------------------------------------------+
  | 1차시  | 컨테이너 명령어 심화(run/ps/stop/rm/logs/exec)                  |
  | 2차시  | Bind Mount 개념 및 실습                                         |
  | 3차시  | Named Volume vs Bind Mount 비교 + 데이터 영속성 심화 실습       |
  +--------+----------------------------------------------------------------+

  # [로컬 VM] 볼륨 바인딩 실습
  mkdir -p ~/html
  echo "<h1>Hello Cloud!</h1>" > ~/html/index.html
  docker run -d -p 8080:80 \
    -v ~/html:/usr/share/nginx/html \
    --name webvol nginx
  curl localhost:8080

  # HTML 수정 후 즉시 반영 확인
  echo "<h1>Updated!</h1>" > ~/html/index.html
  curl localhost:8080

  [3주차] Dockerfile 기반 이미지 빌드 및 Docker Hub 배포
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | Dockerfile 문법(FROM, RUN, COPY, CMD) 및 Docker Hub    |
  |                   | 레지스트리 활용                                        |
  | Killercoda        | Node.js 앱 포함 빈칸 Dockerfile 완성 -> 단일 이미지     |
  | (수업용)          | 빌드 체험                                               |
  | 로컬 VM           | Python Flask 앱 코딩 -> 고유 Dockerfile 빌드 ->         |
  | (과제/복습)       | GitHub 저장소 & Docker Hub에 직접 push (전 과정 수행)   |
  +-------------------+--------------------------------------------------------+

  [3주차 차시 구성]
  +--------+----------------------------------------------------------------+
  | 1차시  | Dockerfile 문법(FROM/RUN/COPY/CMD/EXPOSE)                       |
  | 2차시  | GitHub·Docker Hub 계정 생성 안내 + Flask 앱 빌드 실습            |
  | 3차시  | 이미지 최적화(레이어 캐싱) + Docker Hub push(선택 심화) + 정리   |
  +--------+----------------------------------------------------------------+

  ※ 필수 제출물은 로컬 build+run+curl 확인까지입니다. Docker Hub push는
    계정 가입이 필요한 선택 심화 과제로 두어, 계정 마찰이 핵심 학습(Docker)의
    걸림돌이 되지 않도록 합니다. 2차시 도입부에 GitHub·Docker Hub 가입
    단계별 가이드(스크린샷 포함)를 배포합니다.

  # [로컬 VM] Flask 앱 컨테이너화
  cat > app.py << EOF
  from flask import Flask
  app = Flask(__name__)
  @app.route("/")
  def hello(): return "Hello, Cloud!"
  if __name__ == "__main__": app.run(host="0.0.0.0", port=5000)
  EOF

  cat > Dockerfile << EOF
  FROM python:3.11-slim
  WORKDIR /app
  COPY app.py .
  RUN pip install flask
  EXPOSE 5000
  CMD ["python", "app.py"]
  EOF

  docker build -t {dockerhub_id}/myflask:v1 .
  docker push {dockerhub_id}/myflask:v1

  kubectl get pods -w

  +--------+----------------------------------------------------------------+
  | 1차시  | K8s가 왜 필요한가 + 아키텍처 개관(Control Plane vs Node, 얕게)  |

    kubectl get pods -w로 지켜보도록 안내하세요.

  [5주차] K8s Service와 내부 통신 네트워크
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | ClusterIP / NodePort / LoadBalancer 네트워크 작동 원리 |
  | Killercoda        | NodePort 서비스 생성 -> 특정 포트로 웹 브라우저 접속    |
  | (수업용)          | 결과 퀵 확인                                            |

  # [로컬 VM] 서비스 생성 및 확인
  kubectl expose deployment myapp --type=NodePort --port=80
  kubectl get svc myapp
  minikube service myapp --url

  # Endpoints 확인
  kubectl get endpoints myapp
  # 반복 요청으로 라운드로빈 확인
  for i in $(seq 1 10); do curl $(minikube service myapp --url); done

  [5주차 차시 구성]

  | 1차시  | Service 개념(Pod IP가 불안정한 이유)                            |

  [6주차] ConfigMap, Secret 및 PV/PVC 상태 관리
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | ConfigMap/Secret으로 설정·비밀키 분리 / PV·PVC 영구    |
  |                   | 저장소 아키텍처                                        |
  | Killercoda        | 미리 생성된 Secret 값을 Pod 내부 환경변수로 불러오는    |
  | (수업용)          | 따라하기 실습                                           |
  | 로컬 VM           | VM 물리 디렉토리를 PV로 정의 -> MySQL Pod 연동, VM      |
  | (과제/복습)       | 재부팅 후 DB 레코드 유지 확인(자가 복구 상태 관리 체계) |
  +-------------------+--------------------------------------------------------+

  # [로컬 VM] PV/PVC + MySQL 실습
  # pv.yaml
  apiVersion: v1
  kind: PersistentVolume
  metadata:
    name: mysql-pv
  spec:
    capacity:
      storage: 1Gi
    accessModes: [ReadWriteOnce]
    hostPath:
      path: /data/mysql

  kubectl apply -f pv.yaml
  kubectl apply -f pvc.yaml
  kubectl apply -f mysql-deploy.yaml
  # VM 재시작 후 데이터 유지 확인
  minikube stop && minikube start
  # ※ 재시작 직후 바로 접속하면 "container not found" 오류가 날 수 있습니다.
  #   10~20초 정도 기다린 뒤 아래 명령을 실행하세요.
  kubectl exec -it {mysql-pod} -- mysql -u root -p -e "SHOW DATABASES;"

  [6주차 차시 구성]
  +--------+----------------------------------------------------------------+
  | 1차시  | ConfigMap · Secret 개념(설정·비밀키 분리)                       |
  | 2차시  | PV/PVC 개념(스토리지 추상화) + hostPath 실습                    |
  | 3차시  | MySQL 배포 + 재시작 후 데이터 영속성 검증                       |
  +--------+----------------------------------------------------------------+

  [7주차] Ingress 도메인 라우팅 및 자가 치유(Self-Healing)
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | L7 라우팅 Ingress 컨트롤러 규칙 설계 / K8s 선언적       |
  |                   | 인프라 메커니즘                                        |
  | Killercoda        | Ingress 라우팅 패스 설정 -> /ko, /en 주소별 다른        |
  | (수업용)          | 컨테이너 연결 시뮬레이션                                |
  | 로컬 VM           | hosts 파일 수정으로 가짜 도메인 매핑 -> 실제 서비스     |
  | (과제/복습)       | 도메인 체계 구현, Pod 강제 삭제 후 자가치유 과정 관찰   |
  +-------------------+--------------------------------------------------------+

  ※ 6주차에서 minikube를 재시작한 상태로 이어서 실습하는 경우, 재시작 직후에는
    Ingress 컨트롤러가 아직 준비되지 않아 kubectl apply나 접속이 실패할 수
    있습니다. 10~20초 정도 기다린 뒤 아래 실습을 진행하세요.

  # [로컬 VM] 자가치유 실습

  [12주차 차시 구성]
  +--------+----------------------------------------------------------------+
  | 1차시  | Declarative Pipeline 문법                                       |
  | 2차시  | Jenkinsfile 작성 + 수동 빌드(Build Now) 실습                    |
  | 3차시  | GitHub 연동(Poll SCM) + 자동 빌드 트리거 확인                   |
  +--------+----------------------------------------------------------------+

  // [로컬 VM] Jenkinsfile (Poll SCM 자동 빌드, 검증 완료)
  pipeline {
    agent any
    stages {
      stage("Checkout") {
        steps { git url: "https://github.com/{id}/cloud-lab.git", branch: "main" }
      }
      stage("Build") {
        steps {
          sh "docker build -t {dockerhub_id}/myflask:${BUILD_NUMBER} ."
        }
      }
      stage("Push") {
        steps {
          withCredentials([usernamePassword(credentialsId:"dockerhub-cred",
            usernameVariable:"USER", passwordVariable:"PASS")]) {
            sh "echo $PASS | docker login -u $USER --password-stdin"
            sh "docker push {dockerhub_id}/myflask:${BUILD_NUMBER}"
          }
        }
      }
    }
    post {
      success { echo "빌드 성공: ${BUILD_NUMBER}" }
      failure { echo "빌드 실패" }
    }
  }
  // Jenkins Job 설정 화면에서 Build Triggers -> Poll SCM -> "* * * * *" 체크

  | 1차시  | 파이프라인 아키텍처 설계(전체 흐름도 그려보기)                  |

  # [로컬 VM] 통합 파이프라인 아키텍처
  #
  # [학생 PC 에디터 소스 수정]
  #   └ git push origin main
  #         ↓ (Poll SCM, 매 1분 감지)
  # [로컬 VM Jenkins]
  #  ├ docker build -t {id}/chatbot:${BUILD_NUMBER} .
  #  ├ docker push {id}/chatbot:${BUILD_NUMBER}
  #  └ kubectl set image deployment/chatbot \
  #        web={id}/chatbot:${BUILD_NUMBER}
  #        ↓
  # [Minikube K8s]
  #  └ Rolling Update -> Pod 교체 -> 서비스 운영

  ※ 중요(검증 완료): Jenkins는 Docker 컨테이너로 떠 있어서, Jenkinsfile에서
    kubectl을 곧바로 실행하면 minikube 클러스터에 네트워크로 접속이 안 됩니다
    (별도 도커 네트워크·IP 충돌 문제). 대신 Jenkins에 이미 마운트되어 있는
    /var/run/docker.sock을 이용해, minikube 컨테이너 "안"에 내장된 kubectl을
    그대로 실행하는 방식을 씁니다. Jenkinsfile의 Deploy 단계는 아래처럼
    작성합니다.

  // [로컬 VM] Jenkinsfile Deploy 단계 (docker exec 방식, 검증 완료)
  stage("Deploy") {
    steps {
      sh 'docker exec minikube $(docker exec minikube sh -c "ls /var/lib/minikube/binaries/*/kubectl") \
          --kubeconfig=/etc/kubernetes/admin.conf \
          set image deployment/chatbot web={id}/chatbot:${BUILD_NUMBER}'
      sh 'docker exec minikube $(docker exec minikube sh -c "ls /var/lib/minikube/binaries/*/kubectl") \
          --kubeconfig=/etc/kubernetes/admin.conf \
          rollout status deployment/chatbot --timeout=120s'
    }
  }

  # 배포 확인
  kubectl rollout status deployment/chatbot
  kubectl get pods -w
  curl $(minikube service chatbot --url)

  [14주차] [VM 전용] 부하 시뮬레이션을 통한 Auto-Scaling(HPA) 및 자가 치유 운영

  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+

  |                   | 사용률 감지 -> 챗봇 Pod 실시간 증설(1개->5개) -> 특정    |
  |                   | Pod 강제 Kill 시 자가 치유 라이브 검증                  |
  | 핵심 학습 가치    | 스케일 아웃(Scale-out)과 자가 치유(Self-healing)를      |
  |                   | 터미널 명령으로 직접 유도·관찰 -> 고가용성 인프라       |

  [14주차 차시 구성]
  +--------+----------------------------------------------------------------+
  | 1차시  | HPA 개념 + 오토스케일링 설정(metrics-server, resource requests) |
  | 2차시  | ApacheBench 부하 테스트 실습                                    |
  | 3차시  | 자가치유 실습 + 전체 과정 회고·정리                             |

  ※ 사전 준비: HPA는 metrics-server가 있어야 동작합니다.
  minikube addons enable metrics-server

  ※ 중요(검증 완료): 챗봇 앱이 요청당 실제 연산이 거의 없으면(단순 문자열
    반환 등) ab로 아무리 부하를 줘도 CPU 사용률이 0%에 머물러 스케일업이
    관찰되지 않습니다(2 vCPU VM에서는 ab 자체가 더 큰 부담이 되기도 함).
    아래 두 가지를 함께 적용하세요.
    ① resource request를 낮게 설정(cpu=50m 이하)해 임계값을 쉽게 넘게 함
    ② 챗봇 응답 로직에 약간의 연산(문자열 처리 반복 등)을 넣어 요청당
       CPU 사용이 체감되도록 함

  # [로컬 VM] HPA 설정 및 부하 테스트
  # resource request 설정 (HPA는 request 대비 %로 계산합니다)
  kubectl set resources deployment chatbot -c=web --requests=cpu=50m,memory=64Mi

  # HPA 생성
  kubectl autoscale deployment chatbot \
