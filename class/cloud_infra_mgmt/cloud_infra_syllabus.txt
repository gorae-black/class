※ 본 교과목은 하이브리드(Dual-Track) 전략으로 운영됩니다. 수업 중 빠른 체험은
   Killercoda(브라우저 기반)로, 시간 제약 없는 심화·과제 실습은 학생 개인 로컬
   VM 환경을 활용합니다. 13~14주차 통합 실습은 60분 타임아웃 제한이 있는
   Killercoda를 배제하고 100% 로컬 VM 전용으로 운영합니다.


--------------------------------------------------------------------------------
1. 교과목 개선 배경
--------------------------------------------------------------------------------
  * 기존 교과목은 학생 PC에 멀티노드 Kubernetes 클러스터를 직접 설치하는 방식으로
    운영되어, PC 사양·OS 충돌 등 환경적 변수로 인한 실습 장애가 빈번하였습니다.
  * Docker·K8s 중심의 단일 도구 구성으로 현장에서 요구하는 인프라 자동화(IaC)·
    배포(CI/CD) 역량을 충분히 다루기에 한계가 있었습니다.
  * 이를 개선하여 브라우저 기반 실습(Killercoda)과 로컬 VM 영구 실습 환경을
    병행하는 하이브리드 Dual-Track 전략으로 전면 재구성하였습니다.
  * Terraform(IaC)·Jenkins(CI/CD)를 커리큘럼에 추가하여 클라우드 인프라 자동화
    전 과정을 단계적으로 학습할 수 있도록 확장하였습니다.

2. 핵심 설계 원칙 — 하이브리드 Dual-Track
--------------------------------------------------------------------------------
  +----------------------+-------------------------+---------------------------+----------------------------------------+
  | 트랙                 | 도구                    | 활용 목적                 | 특징                                    |
  +----------------------+-------------------------+---------------------------+----------------------------------------+
  | 수업용                | Killercoda              | 수업 중 빠른 개념 체험,   | 설치 불필요 / 즉시 접속,                |
  | Killercoda 트랙       | (브라우저)              | 핵심 명령어 맛보기        | 60분 제한 있으나 수업 중 활용           |
  +----------------------+-------------------------+---------------------------+----------------------------------------+
  | 심화·과제용           | 개인 로컬 VM            | 시간 제약 없는 심화 실습, | 영구 저장 / 디버깅 몰입 가능,           |
  | 로컬 VM 트랙          | (Multipass/Vagrant)     | 데이터 영구 보존 과제     | 13~14주 통합 실습 전용                  |
  +----------------------+-------------------------+---------------------------+----------------------------------------+

  [3단계 실습 경험 구조]
    따라하기 (완성 코드 실행) -> 수정하기 (파라미터 변경) -> 스스로 작성 (빈칸 완성)


--------------------------------------------------------------------------------
3. [0주차 ~ 1주차 초반] 필수 선행 단계 — 로컬 VM 환경 구성
--------------------------------------------------------------------------------
  목적: 학생들이 학기 전체 및 13~14주차 대통합 실습에서 사용할 개인 로컬 리눅스
        가상머신을 표준화하여 구축한다. PC 사양 및 OS 충돌을 방지하기 위해
        가장 가볍고 검증된 방식을 채택한다.

  +-------------------+--------------------------------------------------------+
  | 항목              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 배포 방식         | 교수자가 작성한 Multipass 환경 구성 스크립트 또는      |
  |                   | Vagrantfile을 1주차 시작 전 LMS에 탑재                 |
  | 학생 활동         | 본인 PC(Windows/Mac)에 가상화 도구 설치 후 명령어      |
  |                   | 한 줄(multipass launch 또는 vagrant up) 실행           |
  | 목표 환경         | Ubuntu 24.04 LTS 기반 개인 실습 VM                      |
  | 환경 검증         | VM 내부 접속 후 docker --version / git --version       |
  |                   | 정상 출력 확인                                         |
  | 필요 사양         | 하드디스크 여유 공간 30GB 이상 / RAM 8GB 이상 권장     |
  +-------------------+--------------------------------------------------------+

  # Windows (Multipass)
  winget install Canonical.Multipass
  multipass launch 24.04 --name cloud-lab --disk 20G --memory 4G
  multipass shell cloud-lab

  # Mac (Multipass)
  brew install multipass
  multipass launch 24.04 --name cloud-lab --disk 20G --memory 4G
  multipass shell cloud-lab

  # VM 내부 검증
  docker --version
  git --version
  echo "VM 환경 구성 완료!"

  [2주차] Docker 컨테이너 생명주기와 데이터 보존
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | 컨테이너 명령어(run, ps, stop, rm), 볼륨 마운트         |
  |                   | (Volume/Bind Mount) 개념                                |
  | Killercoda        | Nginx 컨테이너 빠른 실행 -> 포트 포워딩으로 웹 화면      |
  | (수업용)          | 띄우기 따라하기                                         |
  | 로컬 VM           | 로컬 VM 디렉토리와 Nginx 경로를 볼륨 바인딩, 로컬 HTML  |
  | (과제/복습)       | 수정 시 웹 화면 실시간 변경 확인(데이터 영속성 원리)    |
  +-------------------+--------------------------------------------------------+

  # [로컬 VM] 볼륨 바인딩 실습
  mkdir -p ~/html
  echo "<h1>Hello Cloud!</h1>" > ~/html/index.html
  docker run -d -p 8080:80 \
    -v ~/html:/usr/share/nginx/html \
    --name webvol nginx
  curl localhost:8080

  # HTML 수정 후 즉시 반영 확인
  echo "<h1>Updated!</h1>" > ~/html/index.html
  curl localhost:8080

  [3주차] Dockerfile 기반 이미지 빌드 및 Docker Hub 배포
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | Dockerfile 문법(FROM, RUN, COPY, CMD) 및 Docker Hub    |
  |                   | 레지스트리 활용                                        |
  | Killercoda        | Node.js 앱 포함 빈칸 Dockerfile 완성 -> 단일 이미지     |
  | (수업용)          | 빌드 체험                                               |
  | 로컬 VM           | Python Flask 앱 코딩 -> 고유 Dockerfile 빌드 ->         |
  | (과제/복습)       | GitHub 저장소 & Docker Hub에 직접 push (전 과정 수행)   |
  +-------------------+--------------------------------------------------------+

  # [로컬 VM] Flask 앱 컨테이너화
  cat > app.py << EOF
  from flask import Flask
  app = Flask(__name__)
  @app.route("/")
  def hello(): return "Hello, Cloud!"
  if __name__ == "__main__": app.run(host="0.0.0.0", port=5000)
  EOF

  cat > Dockerfile << EOF
  FROM python:3.11-slim
  WORKDIR /app
  COPY app.py .
  RUN pip install flask
  EXPOSE 5000
  CMD ["python", "app.py"]
  EOF

  docker build -t {dockerhub_id}/myflask:v1 .
  docker push {dockerhub_id}/myflask:v1

  [9주차] Terraform IaC 입문 및 HCL 문법
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | Terraform 아키텍처, HCL 기본 문법, init/plan/apply/    |
  |                   | destroy 핵심 명령어                                    |
  | Killercoda        | Terraform CLI 환경에서 로컬 파일을 코드로 생성·삭제하는 |
  | (수업용)          | 샌드박스 실습                                           |
  | 로컬 VM           | 로컬 VM에 Terraform CLI 연동 -> 로컬 Docker Provider를  |
  | (과제/복습)       | 대상으로 VM 내부 Docker 컨테이너를 코드로 배포·통제하는 |
  |                   | 로컬 IaC 환경 구축                                     |
  +-------------------+--------------------------------------------------------+

  # [로컬 VM] Docker Provider IaC 실습
  # main.tf
  terraform {
    required_providers {
      docker = {
        source  = "kreuzwerker/docker"
        version = "~> 3.0"
      }
    }
  }
  provider "docker" {}
  resource "docker_container" "web" {
    name  = "tf-nginx"
    image = "nginx:latest"
    ports { internal = 80; external = 8080 }
  }

  terraform init
  terraform plan
  terraform apply -auto-approve
  docker ps    # tf-nginx 컨테이너 확인
  terraform destroy -auto-approve

  [12주차] Jenkinsfile 파이프라인 문법 및 GitHub Webhook 연동
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 학습 내용         | Declarative Pipeline 문법, Stage·환경변수·Credential   |
  |                   | 제어                                                   |
  | Killercoda        | 뼈대 Jenkinsfile 빈칸 채우기 -> echo 로그 출력 3단계    |
  | (수업용)          | 실습(스스로 작성) 맛보기                                |
  | 로컬 VM           | GitHub 리포지터리와 로컬 VM Jenkins를 Webhook으로 직접  |
  | (과제/복습)       | 연결 -> 로컬 코드 수정 후 Git Push 시 Jenkins 자동 빌드 |
  |                   | 트리거 확인 -> 로컬 자동화 파이프라인 설계 완성         |
  +-------------------+--------------------------------------------------------+

  // [로컬 VM] Jenkinsfile (GitHub Webhook 자동 빌드)
  pipeline {
    agent any
    stages {
      stage("Checkout") {
        steps { git url: "https://github.com/{id}/cloud-lab.git", branch: "main" }
      }
      stage("Build") {
        steps {
          sh "docker build -t {dockerhub_id}/myflask:${BUILD_NUMBER} ."
        }
      }
      stage("Push") {
        steps {
          withCredentials([usernamePassword(credentialsId:"dockerhub-cred",
            usernameVariable:"USER", passwordVariable:"PASS")]) {
            sh "echo $PASS | docker login -u $USER --password-stdin"
            sh "docker push {dockerhub_id}/myflask:${BUILD_NUMBER}"
          }
        }
      }
    }
    post {
      success { echo "빌드 성공: ${BUILD_NUMBER}" }
      failure { echo "빌드 실패" }
    }
  }

  [13주차] [VM 전용] AI 미니 챗봇 전체 자동 배포 파이프라인 완성
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 실습 미션         | AI 기반 미니 챗봇 웹 앱을 전체 CI/CD 파이프라인으로     |
  |                   | 자동 배포                                              |
  | 구현 아키텍처     | 학생 PC 소스 수정 -> GitHub Push -> 로컬 VM Jenkins     |
  |                   | 감지 -> Docker 이미지 빌드 & Docker Hub Push -> 로컬    |
  |                   | K8s(Minikube) Deployment 자동 교체 배포                |
  | 핵심 학습 가치    | Git(버전관리)+Docker(패키징)+Jenkins(빌드)+K8s(인프라)가|
  |                   | 하나의 유기적 파이프라인 생태계로 연결되는 DevOps       |
  |                   | 엔지니어 경험 체득                                     |
  | 실습 환경         | 100% 로컬 VM (Minikube + Jenkins + Docker)             |
  +-------------------+--------------------------------------------------------+

  # [로컬 VM] 통합 파이프라인 아키텍처
  #
  # [학생 PC 에디터 소스 수정]
  #   └ git push origin main
  #         ↓ (Webhook 트리거)
  # [로컬 VM Jenkins]
  #  ├ docker build -t {id}/chatbot:${BUILD_NUMBER} .
  #  ├ docker push {id}/chatbot:${BUILD_NUMBER}
  #  └ kubectl set image deployment/chatbot \
  #        web={id}/chatbot:${BUILD_NUMBER}
  #        ↓
  # [Minikube K8s]
  #  └ Rolling Update -> Pod 교체 -> 서비스 운영

  # 배포 확인
  kubectl rollout status deployment/chatbot
  kubectl get pods -w
  curl $(minikube service chatbot --url)

  [14주차] [VM 전용] 부하 시뮬레이션을 통한 Auto-Scaling(HPA) 및 자가 치유 운영
  +-------------------+--------------------------------------------------------+
  | 구분              | 내용                                                   |
  +-------------------+--------------------------------------------------------+
  | 실습 미션         | 13주차 운영 중인 챗봇 서버에 가상 대규모 트래픽 폭주    |
  |                   | 상황을 일으켜 클라우드 탄력성을 직접 제어                |
  | 구현 아키텍처     | ApacheBench 부하 툴 트래픽 폭격 -> K8s HPA가 CPU        |
  |                   | 사용률 감지 -> 챗봇 Pod 실시간 증설(1개->5개) -> 특정    |
  |                   | Pod 강제 Kill 시 자가 치유 라이브 검증                  |
  | 핵심 학습 가치    | 스케일 아웃(Scale-out)과 자가 치유(Self-healing)를      |
  |                   | 터미널 명령으로 직접 유도·관찰 -> 고가용성 인프라       |
  |                   | 아키텍처 설계 능력 내재화                               |
  | 실습 환경         | 100% 로컬 VM (Minikube + HPA + ApacheBench)            |
  +-------------------+--------------------------------------------------------+

  # [로컬 VM] HPA 설정 및 부하 테스트
  # HPA 생성
  kubectl autoscale deployment chatbot \
    --cpu-percent=50 --min=1 --max=5
  kubectl get hpa -w &

  # ApacheBench 부하 발생
  ab -n 10000 -c 100 $(minikube service chatbot --url)/

  # Pod 증설 실시간 확인
  kubectl get pods -w

  # 자가 치유 실습
