## [슬라이드 05] 두 가지 능력이 합쳐지면

"우리가 배운 두 가지 능력을 나란히 놓고 볼게요.

**Scale-out**(HPA) — 트래픽이 몰리면 Pod 수를 늘려 부하를 분산합니다. 지난 2차시에 확인했죠.
**자가치유**(Self-Healing) — Pod가 죽으면 즉시 새로 만들어 개수를 유지합니다. 7주차에 배웠습니다.

**(핵심 메시지)** 이 둘이 함께 있으면, '트래픽 폭주'라는 위기와 '장애'라는 위기, 두 가지 모두에서
서비스가 **스스로** 살아남습니다. 사람이 밤새 지키고 있지 않아도요."

## [슬라이드 06] 오늘 검증할 시나리오

"오늘 우리가 확인할 시나리오는 이렇습니다.

부하가 걸려서 Pod가 여러 개로 늘어난 상태에서, 그중 하나를 강제로 `kubectl delete pod`로
삭제합니다. Deployment가 즉시 그 자리를 새 Pod로 채우는지 확인하는데, 이건 7주차에 배운 원리
그대로입니다.

**(관찰 포인트 추가)** 그리고 이 과정 동안 서비스(`curl` 응답)가 완전히 끊기지 않는지도 함께
관찰할 겁니다. Pod 하나가 죽어도 나머지 Pod들이 요청을 처리하고 있으니, 서비스는 계속 살아있어야
정상입니다.

**(7주차 연결)** 7주차에 배운 자가치유가, 오늘은 '부하가 걸린 실전 상황'에서도 그대로 작동하는지
검증하는 거예요."

## [슬라이드 08] VM 실습 — 자가치유 라이브 검증

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "Pod를 여러 개 죽여도 괜찮나요?" | 네, replicas 개수 안에서라면 몇 개를 죽여도 Deployment가 계속 새로 만들어줍니다. 시간이 남는 학생은 여러 개를 연속으로 삭제해보며 자가치유의 견고함을 더 실감해볼 수 있습니다. |
| "부하를 끄면 Pod 개수가 다시 줄어드나요?" | 네, HPA는 CPU 사용률이 목표치 아래로 내려가면 자동으로 Pod 개수를 줄입니다(Scale-in). 다만 즉시 줄지 않고 일정 시간 관찰 후 줄어드는 경우가 많으니, 몇 분 후 `kubectl get hpa`로 다시 확인해보라고 안내하세요. |
| "이번 학기 실습을 나중에 복습하려면 어떻게 하나요?" | 각 주차 폴더의 04_템플릿코드와 06_로컬VM_과제복습(13~14주차는 05_로컬VM_통합실습_전용) 자료를 참고하면 됩니다. VM을 그대로 보관해두면 실습 환경을 다시 켜서 복습할 수 있다는 것도 안내해주세요. |
