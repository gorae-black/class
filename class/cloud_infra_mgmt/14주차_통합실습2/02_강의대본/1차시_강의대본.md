## [슬라이드 05] 트래픽은 항상 일정하지 않다

"1주차에서 클라우드의 5대 특징을 배웠던 거 기억하시나요? 그중 하나가 '신속한 확장성'이었습니다.

**(시나리오 제시)** 자, 13주차에 만든 챗봇이 갑자기 인기를 끌었다고 상상해보세요. 지금 Pod가
1개뿐이라면, 사용자가 몰릴 때 응답이 느려지거나 심하면 다운될 수 있습니다.

**(반대 시나리오)** 그렇다고 처음부터 Pod를 5개씩 띄워두면 어떨까요? 평소 트래픽이 적을 땐 자원을
낭비하는 셈이 됩니다.

**(해결책 제시)** 이 딜레마를 해결하는 게 **HPA**, Horizontal Pod Autoscaler입니다. CPU 사용률
같은 지표를 보고, 필요할 때만 자동으로 Pod 개수를 늘리거나 줄이는 K8s 기능이에요."

## [슬라이드 07] HPA의 판단 과정

"HPA의 동작은 세 단계로 이뤄집니다.

먼저 **metrics-server**가 각 Pod의 실제 CPU 사용률을 주기적으로 수집합니다. 그다음 **HPA**가
'설정한 목표치'(예: 50%)와 '실제 사용률'을 비교합니다. 목표치를 넘으면 replicas를 늘리고
(**Scale-out**), 사용률이 줄어들면 다시 Pod 개수를 줄입니다(**Scale-in**).

**(4주차 연결)** 4주차에서 배운 선언적 관리, 기억하시나요? HPA도 똑같은 원리예요 — '목표'와
'현재 상태'를 계속 비교해서 스스로 맞춰나가는 겁니다. 다만 이번엔 그 목표가 '고정된 replicas 수'가
아니라 'CPU 사용률'이라는 게 다릅니다."

---

## [슬라이드 08] 사전 준비 ① — metrics-server

"HPA를 쓰려면 준비물이 두 가지 필요합니다. 첫 번째는 **metrics-server**입니다.

HPA는 CPU 사용률을 알아야 판단할 수 있는데, 기본 minikube에는 이 수집 기능이 꺼져있습니다.
`minikube addons enable metrics-server`로 활성화해야 해요.

활성화가 잘 됐는지는 `kubectl top pods` 명령으로 확인할 수 있습니다. 실제 Pod별 CPU 사용률
숫자가 보이면 준비가 끝난 겁니다."

---

## [슬라이드 09] 사전 준비 ② — resource request

"두 번째 준비물은 **resource request**입니다.

**(핵심 개념)** HPA는 '실제 사용률'을 '요청한 양(request) 대비 %'로 계산합니다. 6주차에 PV/PVC를
배울 때도 request라는 말을 썼었는데, 여기서는 '컨테이너가 최소한 보장받는 자원의 양'을 의미합니다.

설정 명령어는 이렇습니다.
```
kubectl set resources deployment chatbot -c=web --requests=cpu=50m,memory=64Mi
```

**(검증된 주의사항 강조)** 여기서 정말 중요한 포인트가 있습니다 — request를 너무 높게 잡으면,
실제 사용률이 그 대비 %로 계산되기 때문에 목표치(50%)를 넘기기 어려워져서 스케일업이 관찰되지
않습니다. 그래서 오늘은 일부러 `cpu=50m`처럼 **낮게** 설정합니다."

---

## [슬라이드 10] HPA 생성 명령어

"이제 실제로 HPA를 만드는 명령어입니다.

```
kubectl autoscale deployment chatbot --cpu-percent=50 --min=1 --max=5
```

CPU 사용률이 50%를 넘으면 Pod를 늘리고, 최소 1개에서 최대 5개 사이에서 조정하겠다는 뜻입니다.

`kubectl get hpa`를 실행하면 TARGET(현재/목표 %)과 REPLICAS(현재 Pod 수)를 실시간으로 확인할 수
있습니다."

---

## [슬라이드 11] VM 실습 — metrics-server + HPA 설정

"자, 직접 해보겠습니다.

`minikube addons enable metrics-server`를 실행한 뒤 `kubectl top pods`로 수집이 되는지
확인하세요.

`kubectl set resources`로 chatbot의 resource request를 낮게 설정하세요.

`kubectl autoscale deployment chatbot --cpu-percent=50 --min=1 --max=5`를 실행하고, `kubectl get
hpa`로 HPA가 생성되고 TARGET 값이 보이는지 확인하세요."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "kubectl top pods에 값이 안 보여요" | metrics-server가 완전히 준비되기까지 1분 정도 걸릴 수 있습니다. 잠시 기다린 후 다시 시도하도록 안내하세요. 그래도 안 되면 `kubectl get pods -n kube-system`으로 metrics-server Pod가 Running인지 확인합니다. |
| "request를 낮게 잡으면 실제로 서비스에 문제가 없나요?" | request는 '최소 보장량'일 뿐, Pod가 그 이상 자원을 쓸 수 없다는 뜻은 아닙니다(별도로 limit을 설정하지 않는 한). 오늘 실습에서는 HPA 동작을 쉽게 관찰하기 위한 실습용 설정이라는 점을 안내하세요. |
| "min=1, max=5는 무슨 기준으로 정하나요?" | 실무에서는 예상 트래픽과 서버 자원 한계를 고려해 정합니다. 이 과목에서는 우리 VM의 자원 제약(2 vCPU)을 고려해 5개를 상한으로 잡았습니다. |
