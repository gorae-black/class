## [슬라이드 04] Part 1 섹션 표지 — 왜 random_pet을 쓰는가

"오늘 실습에 들어가기 전에, 예제 선택 이유부터 짚고 가겠습니다. 이거 놓치면 실습에서 바로 오류를

## [슬라이드 05] 9주차 예제를 그대로 쓰면 생기는 문제

"혹시 '9주차에 썼던 docker_container 예제를 그대로 쓰면 되지 않을까?'라고 생각하신 분 있나요?

**(직접 실험해서 보여주기)** 실제로 해보면 이렇게 됩니다 — 원격으로 apply를 실행하면 \"Cannot
connect to the Docker daemon\"이라는 오류가 납니다.

**(원인 설명)** 왜 이런 오류가 날까요? 1차시에서 배운 대로, 원격 실행은 Terraform Cloud의
**서버**에서 이뤄집니다. 그런데 docker_container 리소스는 '로컬 Docker 데몬'에 접속해야 컨테이너를
만들 수 있어요. Terraform Cloud 서버는 여러분의 VM Docker에 접근할 방법이 전혀 없으니, 이 오류가
나는 게 당연한 겁니다.

**(해결책)** 그래서 오늘은 로컬 인프라에 전혀 의존하지 않는 `random` Provider의 `random_pet`
리소스를 사용합니다. 실제 인프라를 만들진 않지만, 무작위 이름을 생성하는 이 리소스가 '원격 실행이
정말로 되는지' 검증하기에 딱 맞습니다."

## [슬라이드 07] 하이브리드 GitOps 흐름

"오늘 완성할 전체 흐름은 이렇습니다.

로컬 VM에서 코드를 작성하고 GitHub 저장소에 push합니다. Terraform Cloud 워크스페이스를 그 GitHub
저장소와 연결(VCS 연동)해두면, push가 감지될 때마다 Terraform Cloud가 자동으로 plan을 실행하고,
확인 후 apply할 수 있습니다.

**(용어 설명)** 이렇게 코드 변경이 Git을 매개로 자동으로 인프라에 반영되는 방식을 **GitOps**라고
부릅니다. 로컬 작성 + 원격 실행이 결합된 형태라 '하이브리드 GitOps'라고 표현했습니다."

## [슬라이드 08] main.tf — random_pet 예제

"오늘 쓸 코드는 이렇습니다.

```hcl
resource "random_pet" "lab" {
  length = 2
}
output "pet_name" {
  value = random_pet.lab.id
}
```

`random_pet`은 무작위 형용사+동물 이름을 만들어주는 리소스입니다. `length = 2`는 단어를 2개
조합한다는 뜻이에요. `apply`가 끝나면 `terraform output`으로 `pet_name = \"grand-quail\"`처럼
생성된 이름을 확인할 수 있습니다."

---

## [슬라이드 09] 실행 흐름 재확인

"실행 흐름을 다시 정리하면 이렇습니다.

`terraform init` — 2차시에 만든 cloud 블록의 워크스페이스에 연결합니다.
`terraform apply` — **이 순간 실제 연산은 Terraform Cloud 서버에서 실행**됩니다.
`terraform output` — 결과값(`pet_name`)을 확인합니다.

그리고 마지막으로, `app.terraform.io`에 접속해서 워크스페이스의 **Runs** 탭을 열어보면, 방금 실행한
기록이 그대로 남아있는 걸 확인할 수 있습니다."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "그러면 Terraform Cloud로는 로컬 Docker를 절대 못 다루나요?" | 이번 학기 실습 범위에서는 그렇습니다. 실무에서는 'Terraform Cloud Agent'라는 별도 구성으로 사설 네트워크 안의 자원에 원격 실행이 접근하게 만들 수도 있지만, 이 과목의 범위를 넘어서는 고급 주제입니다. |
| "GitHub 연동 없이 CLI로만 apply해도 원격 실행이 되나요?" | 네, 2차시에서 배운 cloud 블록만 설정되어 있으면 `terraform apply`를 CLI에서 실행해도 실제 연산은 원격에서 이뤄집니다. GitHub 연동은 '코드 변경을 자동으로 감지해서 실행'하는 추가 기능입니다. |
| "random_pet 같은 예제가 실무에서도 쓰이나요?" | random_pet 자체는 실습·테스트용입니다. 하지만 실무에서는 AWS EC2, GCP Compute Engine 같은 실제 클라우드 리소스를 같은 방식(코드 작성 → GitHub push → 원격 apply)으로 다룹니다. 오늘 배운 흐름의 구조 자체는 실무와 동일합니다. |
| "Killercoda 링크가 언제 만들어졌는지에 따라 제한 시간이 달라지나요?" | 아니요. 60분 타이머는 링크 생성 시점이 아니라 각 학생이 시나리오를 '시작'한 순간부터 개별적으로 시작됩니다. 링크 자체는 만료되지 않으므로, 비동기로 늦게 접속하거나 여러 학생이 같은 링크를 써도 각자 독립된 환경과 타이머를 받습니다. |
