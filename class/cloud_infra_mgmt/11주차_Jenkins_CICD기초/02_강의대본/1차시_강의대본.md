## [슬라이드 05] 지금까지 우리는 배포를 손으로 했다

"3주차에서 Docker Hub에 이미지를 push했던 것 기억하시나요? `docker build` → `docker push`, 전부
손으로 명령어를 입력했습니다.

**(문제 제기)** 그런데 코드를 한 줄 고칠 때마다 이 과정을 매번 반복해야 한다면 어떨까요? 게다가
여러 사람이 함께 작업한다면, 누가 언제 무엇을 빌드하고 배포했는지 추적하기도 어려워집니다.

**(해결책 제시)** 이 문제를 해결하는 게 **CI/CD**입니다. CI는 Continuous Integration(지속적
통합), CD는 Continuous Deployment(지속적 배포)의 줄임말이에요."

---

## [슬라이드 06] CI와 CD를 나눠서 이해하기

"CI와 CD를 각각 나눠서 볼게요.

**CI(지속적 통합)** — 코드가 바뀔 때마다 **자동으로** 빌드하고 테스트해서, 문제가 있으면 빨리
발견합니다.

**CD(지속적 배포)** — 그렇게 빌드된 결과물을 **자동으로** 실제 서비스 환경까지 배포합니다.

**(핵심 요약)** 이 둘을 자동으로 이어주는 파이프라인을 만드는 게 오늘 배울 CI/CD 도구의 역할입니다.
'코드 수정 → 빌드 → 배포'라는 흐름에서, 사람이 하던 반복 작업을 도구가 대신 해주는 거예요."

## [슬라이드 08] Jenkins의 핵심 구성요소

"Jenkins의 구조를 네 가지 개념으로 정리하면 이렇습니다.

**Jenkins 서버** — 웹 UI를 통해 자동화 작업을 관리하는 자동화 서버 자체입니다.
**Job(작업)** — '무엇을, 어떻게 빌드·배포할지'를 정의한 단위입니다.
**Build** — Job이 실제로 한 번 실행된 결과입니다(성공/실패, 로그, 산출물이 남습니다).
**Plugin** — Git 연동, Docker 연동처럼 Jenkins의 기능을 확장해주는 부품입니다.

**(그림 설명)** 화면의 그림을 보시면, Jenkins 서버 아래에 Job이 있고, 그 Job이 실행될 때마다 Build
#1, #2, #3처럼 실행 이력이 쌓입니다."

## 강의자 참고 — 예상 질문 & 답변

| 예상 질문 | 답변 가이드 |
|---|---|
| "Jenkins 말고 다른 CI/CD 도구는 없나요?" | GitHub Actions, GitLab CI 같은 도구도 널리 쓰입니다. Jenkins는 가장 오래되고 널리 쓰이는 오픈소스 CI/CD 도구이며, 이 과목에서는 직접 서버를 설치·운영해보는 경험을 위해 Jenkins를 선택했습니다. |
| "CI만 하고 CD는 안 해도 되나요?" | 네, 실무에서도 CI까지만 자동화하고 최종 배포는 사람이 승인 후 수동으로 하는 경우가 많습니다. 이 과목의 13주차 실습에서는 CI와 CD를 모두 자동화하는 전체 파이프라인을 경험해봅니다. |
| "Plugin은 누가 만드나요?" | Jenkins 커뮤니티와 각 도구 개발사(GitHub, Docker 등)가 만들어 공개 저장소에 배포합니다. Jenkins 관리 화면에서 필요한 Plugin을 검색해 설치할 수 있습니다. |
| "Killercoda 링크가 언제 만들어졌는지에 따라 제한 시간이 달라지나요?" | 아니요. 60분 타이머는 링크 생성 시점이 아니라 각 학생이 시나리오를 '시작'한 순간부터 개별적으로 시작됩니다. 링크 자체는 만료되지 않으므로, 비동기로 늦게 접속하거나 여러 학생이 같은 링크를 써도 각자 독립된 환경과 타이머를 받습니다. |
